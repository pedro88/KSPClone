# M4 — Comms & ground control — code-ready tasks

> Expanded from roadmap.md slices 4.1–4.3. Each ticket cites the spec requirement(s) it satisfies. Terms in *italics* are in CONTEXT.md.

## Slice 4.1 — Link & blackout gating

### M4-T01 — Build the CommNet graph from relays and ground stations
- **Satisfies:** COMMS-1
- **Depends on:** —
- **Blocked by:** —
- **Intent:** Stand up the data model and per-tick assembly of the *CommNet* graph (ground stations + relay-capable vessels) that everything else in M4 evaluates *links* against. No occlusion or range logic yet — just the nodes, their world positions, and their advertised antenna capability.
- **Touches:** `CommNetGraph` (engine-agnostic C# in Sim core), `CommNode` (ground station / relay vessel), `AntennaSpec` (range/power, relay-capable flag), `GroundStationRegistry` (home-body stations from world config).
- **Steps:**
  1. Define `CommNode` carrying source kind (GroundStation | RelayVessel | EndpointVessel), world position resolved from *on-rails* `position(game-time)` or active-physics transform, and an `AntennaSpec`.
  2. Define `AntennaSpec` (max range, relay-capable boolean, power rating) sourced from part data; aggregate per-vessel into one effective spec.
  3. Build `GroundStationRegistry` seeded from static world config (home body surface stations) as always-on nodes.
  4. Assemble `CommNetGraph` each evaluation tick from the registry plus all vessels advertising a relay/endpoint antenna, keyed by stable vessel/station ID.
  5. Run graph assembly on the fixed-step scheduler (Constitution Art. 2), not a render callback; throttle assembly to the snapshot cadence, not every 60 Hz physics tick.
- **Acceptance:** A unit test with two ground stations and three vessels produces a `CommNetGraph` with exactly five nodes, each at the position returned by its propagator/transform at a given *game-time*.
- **Est:** M

### M4-T02 — Evaluate link reachability with range and celestial-body occlusion
- **Satisfies:** COMMS-1
- **Depends on:** M4-T01
- **Blocked by:** —
- **Intent:** Decide, per vessel per tick, whether a *link* to the *CommNet* exists by finding any reachable path to a ground station through relays, where each edge must pass both a range check and a line-of-sight check (no celestial body between the two endpoints).
- **Touches:** `LinkEvaluator`, `OcclusionTest` (ray vs body-sphere), `CommNetGraph` (edge set), `LinkResult` (linked bool + path + signal margin).
- **Steps:**
  1. Implement `OcclusionTest`: for an endpoint pair, ray/segment vs each celestial body modeled as a sphere (body radius); edge is occluded if the segment intersects any sphere other than at its own endpoints' bodies.
  2. Implement the per-edge range check from the two nodes' `AntennaSpec` (combined power vs distance) producing a pass/fail plus a signal margin.
  3. Build the reachability pass: an edge is usable if range-pass AND not occluded AND at least one endpoint is relay-capable (or is a ground station / the evaluated endpoint); run a graph traversal (BFS/Dijkstra on signal margin) from each endpoint vessel to any ground station.
  4. Emit a `LinkResult` per vessel: linked = path found; include the winning path and weakest-edge margin for later UI.
  5. Reuse patched-conics `position(t)` for node positions so evaluation is valid under *on-rails warp* without stepping (ORBIT-2).
- **Acceptance:** In a scripted scene, a vessel in direct line of a ground station reports linked; when its orbit carries it behind the body (segment intersects the body sphere), the next evaluation reports not-linked; routing through a relay positioned with clear line-of-sight restores linked.
- **Est:** L

### M4-T03 — Maintain and replicate authoritative per-vessel link status with hysteresis
- **Satisfies:** COMMS-1
- **Depends on:** M4-T02
- **Blocked by:** —
- **Intent:** Turn raw `LinkResult`s into a stable, server-authoritative link-status field on each vessel, replicated to clients, with debounce/hysteresis so a vessel skimming an occlusion edge does not chatter linked/unlinked every tick.
- **Touches:** `LinkStatusComponent` (per-vessel: Linked | Blackout, since-game-time, path), `LinkStatusService`, snapshot/replication layer, `LinkStatusView` (client read-side).
- **Steps:**
  1. Add `LinkStatusComponent` to vessel state as authoritative server state (NET-1); never client-authored.
  2. In `LinkStatusService`, apply hysteresis: require N consecutive evaluations (or a small game-time dwell) before flipping state to suppress edge chatter.
  3. Add link status to the vessel snapshot payload emitted at 20–30 Hz; clients render it read-only via `LinkStatusView`.
  4. Write the link state transition through to durable storage as a meaningful event (PERSIST-2 alignment) and re-derive it on restart rather than trusting a stale row.
  5. Expose a server query `IsLinked(vesselId)` consumed by the blackout gate (M4-T04).
- **Acceptance:** A client UI shows a live linked/blackout indicator that matches the server within one snapshot interval; a vessel grazing a body's limb does not toggle status more than once per configured dwell window in a logged 60 s test.
- **Est:** M

### M4-T04 — Gate networked capabilities on link state via a single blackout gate
- **Satisfies:** COMMS-2, COMMS-3
- **Depends on:** M4-T03
- **Blocked by:** In-game comms modality (spec, Slice 4.2) — the comms-channel capability is gated here but only fully testable once M4-T08 lands; gate the others now.
- **Intent:** Introduce one chokepoint that disables a vessel's *networked capabilities* (map-assist, maneuver-node computation, cross-vessel telemetry, science transmission, in-game comms) whenever it is in *blackout*, while leaving onboard crew hand-flight untouched (Constitution Art. 5). The comms-channel capability is registered with the gate now even though its modality is TBD.
- **Touches:** `BlackoutGate`, `NetworkedCapability` (enum/flags: MapAssist, NodeCompute, CrossVesselTelemetry, ScienceTransmit, InGameComms), capability consumers (map view, node computer, telemetry feed, science system, comms channel).
- **Steps:**
  1. Define the `NetworkedCapability` set and route every networked feature through `BlackoutGate.IsAvailable(vesselId, capability)`.
  2. Implement the gate as: available ⇔ `LinkStatusService.IsLinked(vesselId)`; in blackout all capabilities return unavailable.
  3. Wire each consumer to query the gate before acting (e.g. node computer refuses to compute, science system refuses to transmit, telemetry feed stops cross-vessel updates).
  4. Explicitly keep the pilot/engineer/navigator *direct input* path (station hand-flight on instruments) OUTSIDE the gate — it must never consult `BlackoutGate` (Constitution Art. 5, COMMS-3).
  5. Register the InGameComms capability with the gate as a stub consumer so M4-T08 only has to attach the real channel.
- **Acceptance:** With link forced to blackout, the maneuver-node computer, map-assist, cross-vessel telemetry, and science transmission all report unavailable; meanwhile a seated pilot's throttle/attitude inputs still move the vessel in a flight test. Restoring link re-enables all gated capabilities.
- **Est:** M

### M4-T05 — Make a crewless, no-link vessel dead-stick (uncommandable)
- **Satisfies:** COMMS-4, COMMS-3
- **Depends on:** M4-T04
- **Blocked by:** —
- **Intent:** When a vessel has no onboard crew AND no *link*, reject all commands so it is dead-stick, while a vessel that has either crew OR a link remains commandable. This is the inverse-corner complement to Art. 5: no local crew and no network means no authority at all.
- **Touches:** `CommandabilityGate`, station/automation input path, `BlackoutGate`, crew-presence query.
- **Steps:**
  1. Implement `CommandabilityGate.IsCommandable(vesselId)` = (hasOnboardCrew) OR (IsLinked). Onboard crew means an occupied station with a human, not *automation fallback* alone on an unlinked probe.
  2. Route remote/ground and *automation fallback* command application for unmanned vessels through this gate; if uncommandable, drop the command and surface a "no signal, no crew" reason to the issuer.
  3. Ensure a crewed-but-blackout vessel stays commandable by its local crew (defers to Art. 5 hand-flight path, never gated).
  4. Ensure a linked-but-crewless probe stays commandable from ground/remote (link present).
  5. Distinguish automation-fallback-on-a-crewed-vessel (allowed) from automation on an unmanned unlinked probe (must be inert).
- **Acceptance:** An unmanned probe driven into *blackout* ignores all incoming commands (logged as dropped); the same probe, once it regains link, accepts commands again; a crewed vessel in blackout still responds to its seated crew.
- **Est:** M

## Slice 4.2 — In-game comms (resolve modality + P-2 first)

### M4-T06 — Decide and document in-game comms modality and transport (design spike)
- **Satisfies:** COMMS-2
- **Depends on:** M4-T03
- **Blocked by:** In-game comms modality (spec open item, line 99 of spec.md) and P-2 comms transport (plan.md). This ticket resolves both; downstream tickets unblock once it lands.
- **Intent:** Close the two open decisions that block real in-game comms: which modality(ies) ship first (text / voice / data-sharing / pings / shared markers) and whether the channel reuses the replication channel or runs a separate transport (P-2). Produce a written spec amendment + ADR so M4-T07/T08 are code-ready. Until resolved, M4-T08 builds against the chosen interface.
- **Touches:** spec.md (resolve open item, tighten COMMS-2 testability), plan.md (resolve P-2), new ADR `00xx-in-game-comms-transport.md`, `ICommsChannel` interface sketch.
- **Steps:**
  1. Enumerate candidate modalities and pick the first-cut set (recommend: text + pings/shared markers as the minimal *in-game comms*; voice deferred or routed out-of-band per CONTEXT blackout note).
  2. Decide transport (P-2): reuse the authoritative snapshot/replication channel vs a separate channel; record bandwidth, ordering, and blackout-gating implications of each.
  3. Define the `ICommsChannel` server interface (send, receive, per-vessel routing) the implementation will target.
  4. Amend spec.md to make COMMS-2 testable for the chosen modality; write the ADR with rejected alternatives.
  5. Confirm the channel is a gated `NetworkedCapability` (M4-T04) so it is cut in blackout by construction.
- **Acceptance:** spec.md no longer lists "in-game comms modality" as open; plan.md P-2 is marked resolved with a chosen transport; an ADR exists; `ICommsChannel` is specified well enough that M4-T08 needs no further design questions.
- **Est:** M

### M4-T07 — Route comms messages over the CommNet topology (server-side)
- **Satisfies:** COMMS-2
- **Depends on:** M4-T06, M4-T02
- **Blocked by:** Resolved once M4-T06 lands (modality + P-2).
- **Intent:** Implement server-authoritative comms routing so a message between players/vessels is only deliverable when a *CommNet* path exists between the relevant endpoints, reusing the same link reachability the gate uses. A message with no path is undeliverable, not queued-forever.
- **Touches:** `CommsRouter`, `ICommsChannel` (impl), `LinkEvaluator`/`CommNetGraph` (path query), `BlackoutGate`.
- **Steps:**
  1. Implement `CommsRouter` on the server consuming `ICommsChannel` sends; resolve sender and recipient endpoints to `CommNode`s.
  2. For each send, query the existing reachability path (M4-T02) between endpoints; deliver only if a path exists and both endpoints pass `BlackoutGate` for InGameComms.
  3. Apply the transport decision from M4-T06 (shared vs separate channel) for the actual wire delivery.
  4. Define behavior for no-path sends per M4-T06 (drop with "no signal" feedback; no infinite queue that would fake delivery — Constitution Art. 4 "nothing is faked").
  5. Keep all routing authority on the server (NET-1); clients never self-authorize delivery.
- **Acceptance:** Two linked vessels exchange a comms message that the server delivers; when one vessel loses link (occlusion), a subsequent send from it is rejected/dropped with a "no signal" result, and no message is silently delivered.
- **Est:** M

### M4-T08 — Attach the in-game comms channel to the blackout gate, end to end
- **Satisfies:** COMMS-2
- **Depends on:** M4-T07, M4-T04
- **Blocked by:** Resolved once M4-T06 lands.
- **Intent:** Wire the real comms channel into the client and prove the full slice: players can use *in-game comms* only while linked, and it is cut the instant a participant enters *blackout* — replacing the M4-T04 stub consumer.
- **Touches:** `BlackoutGate` (InGameComms consumer), `CommsRouter`, client `CommsView` (compose/receive UI for chosen modality), `LinkStatusView`.
- **Steps:**
  1. Replace the InGameComms stub registered in M4-T04 with the real `CommsRouter`-backed channel.
  2. Build the client `CommsView` for the chosen modality (text/pings/markers from M4-T06): compose, send, receive, and a disabled state shown while in blackout.
  3. On a participant's link loss, the gate disables send/receive for that endpoint; on restore, re-enable — driven by replicated `LinkStatusComponent`, not client guess.
  4. Ensure out-of-band voice (e.g. Discord) is untouched — only the in-game channel is gated (CONTEXT blackout note).
  5. Add an integration test driving two clients through link → comms exchange → forced blackout → comms cut → restore.
- **Acceptance:** Two connected clients exchange in-game comms while linked; forcing one into blackout disables that client's in-game comms UI and blocks delivery within one snapshot interval; restoring link re-enables it; out-of-band voice is unaffected (not gated).
- **Est:** M

## Slice 4.3 — Ground control (resolve COMMS-6 first)

### M4-T09 — Build the ground-control read-model (map, telemetry, planning, relay view)
- **Satisfies:** COMMS-5
- **Depends on:** M4-T03
- **Blocked by:** —
- **Intent:** Provide the data backing for *ground control* mode: a read-only projection of world state (vessels, orbits, telemetry, CommNet topology) any player can open anytime, assembled server-side and replicated. No commands flow through this model.
- **Touches:** `GroundControlReadModel`, `GroundControlView` (client), `CommNetGraph` (relay topology), telemetry/orbit projections, snapshot/replication layer.
- **Steps:**
  1. Define `GroundControlReadModel`: vessel list with *on-rails*/active state, orbital elements, link status, and CommNet node/edge topology for relay-network management display.
  2. Assemble it server-side from authoritative state (NET-1) and replicate as a read-only feed; do not expose any mutation path through it.
  3. Build `GroundControlView` client UI: map view, per-vessel telemetry, mission/transfer planning surface, and a relay-network map.
  4. Allow any player to open the view at any time regardless of station occupancy (it is off-vessel and holds no authority).
  5. Reuse closed-form `position(t)` for map/planning projections so the view is consistent under warp without physics stepping.
- **Acceptance:** Any connected player can open ground-control mode and see live map, telemetry, and the relay-network topology; the view updates from snapshots and exposes no control affordance that mutates world state.
- **Est:** M

### M4-T10 — Enforce that ground control holds no physics authority
- **Satisfies:** COMMS-5
- **Depends on:** M4-T09
- **Blocked by:** —
- **Intent:** Guarantee by construction that *ground control* mode can never command physics or write authoritative vessel state — it is strictly read-only — so opening it can never conflict with seated crew (Constitution Art. 1, Art. 6).
- **Touches:** `GroundControlReadModel` (read-only contract), input routing, server command validation, station authority map.
- **Steps:**
  1. Ensure ground-control mode has no station occupancy and no entry in the input → station routing map; it cannot acquire a physics-authoritative channel.
  2. Add a server-side guard rejecting any physics/vessel-state command whose origin is the ground-control surface.
  3. Make the read-model a one-way projection type (no setters / no command DTOs) so misuse fails at compile time where possible.
  4. Verify map/planning edits (e.g. drafting a transfer) stay client-local previews until handed to a real station (Navigator) that owns maneuver nodes (CREW-1) — ground control itself never commits them.
- **Acceptance:** An automated test issues a physics command tagged with a ground-control origin and the server rejects it; a player in ground-control mode concurrently with a seated pilot cannot alter the vessel's thrust/attitude or any authoritative state.
- **Est:** S

### M4-T12 — Implement ground-control assignment and program bonuses
- **Satisfies:** COMMS-6
- **Depends on:** M4-T11, M4-T09
- **Blocked by:** Resolved once M4-T11 lands.
- **Intent:** Implement the assignment mechanism and bonus application decided in M4-T11: a player may opt into a ground-control assignment which grants the defined program-wide bonuses, with the system functioning normally when unassigned. Server-authoritative, read-only with respect to physics (assignment grants bonuses, never physics authority — M4-T10 still holds).
- **Touches:** `GroundControlAssignment`, `GroundControlBonusModifier`, `GroundControlReadModel`, capability/assist consumers, replication layer.
- **Steps:**
  1. Implement the assignment claim/release mechanism from M4-T11 as authoritative server state (NET-1), replicated so all players see who is assigned.
  2. Implement `GroundControlBonusModifier` applying the defined bonuses to their target systems (e.g. telemetry refresh rate, assist quality) while assigned.
  3. Apply bonuses at the shared-program scope (Constitution Art. 8); revert to baseline on release/disconnect (no reserved post, mirroring CREW-5 spirit).
  4. Ensure assignment never grants physics authority — bonuses only affect *networked capability* quality/read-model, never command authority (M4-T10).
  5. Add tests for assign → bonus active, unassigned → baseline, disconnect → revert.
- **Acceptance:** Assigning a player to ground control produces the COMMS-6-defined bonuses (observable, e.g. measurably higher telemetry refresh / assist quality) program-wide; unassigning or disconnecting reverts to baseline; assignment never enables any physics command.
- **Est:** M
