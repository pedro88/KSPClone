# M3 ‚Äî Collaborative VAB ‚Äî code-ready tasks

> Expanded from roadmap.md slices 3.1‚Äì3.3. Each ticket cites the spec requirement(s) it satisfies. Terms in *italics* are in CONTEXT.md.

## Slice 3.1 ‚Äî Design model + edit op-log

### M3-T01 ‚Äî Define the Design part-tree data model
- **Satisfies:** BUILD-1
- **Depends on:** ‚Äî
- **Intent:** Establish a *Design* as a *part tree* of *parts* attached parent-child, with no world state, living entirely in the engine-agnostic Design system (separate path, Constitution Art. 7).
- **Touches:** `Design` (entity: design id, name, root node ref, op-log seq cursor), `PartNode` (stable node id, part-type id, parent node id, attach-point ref, local transform), `PartTree` (container + parent/child index), `PartCatalog` (part-type defs: id, attach points, mass, dry/resource slots ‚Äî read-only). All under a `Construction` assembly with **zero** references to flight/physics/replication assemblies.
- **Steps:**
  1. Model `PartNode` with a server-assigned stable node id (monotonic per Design) that never changes for the life of the node, plus parent node id, attach-point key, and local transform relative to parent.
  2. Model `PartTree` holding the node set with a parent‚Üíchildren index and a root node; expose lookup-by-node-id, subtree enumeration, and ancestor walk.
  3. Model `Design` as the aggregate root (design id, display name, root `PartNode` id, current applied op sequence number) with no fields for position/velocity/resources/crew.
  4. Add an assembly-definition boundary (asmdef) so `Construction` cannot reference `Flight`/`Physics`/`Replication`; this dependency rule is the machine-checkable form of Art. 7.
- **Acceptance:** A `Design` can be constructed in memory and queried (root, children, subtree of any node) in a unit test that links only the `Construction` assembly; attempting to add a reference from `Construction` to the flight/physics assembly fails to compile.
- **Est:** M

### M3-T02 ‚Äî Define edit op types and pure tree-mutation application
- **Satisfies:** BUILD-1, BUILD-2
- **Depends on:** M3-T01
- **Intent:** Define the three *edit op* kinds (add/remove/move part) targeting nodes by stable id, and a pure function that applies one op to a `PartTree`, producing the deterministic next tree state.
- **Touches:** `EditOp` (sealed hierarchy: `AddPartOp{newNodeId, partTypeId, parentNodeId, attachPoint, localTransform}`, `RemovePartOp{nodeId}`, `MovePartOp{nodeId, newParentNodeId, newAttachPoint, newLocalTransform}`), `EditOpResult` (Applied | Rejected{reason}), `PartTreeMutator.Apply(PartTree, EditOp) -> EditOpResult`.
- **Steps:**
  1. Define each op as immutable data referencing nodes only by stable id (never by index or object ref).
  2. Implement `Apply` as a pure function: `Add` validates parent exists and attach point is free; `Remove` removes the node and its whole subtree; `Move` re-parents a node (and its subtree) and rejects if the new parent is inside the moved subtree (cycle guard).
  3. Define rejection reasons as an enum (`UnknownNode`, `AttachPointOccupied`, `WouldCreateCycle`, `RemoveRoot`) so callers and tests can assert the exact cause.
  4. Guarantee determinism: `Apply` does no I/O, no clock reads, no randomness ‚Äî same (tree, op) always yields the same result.
- **Acceptance:** Unit tests show each op kind mutates the tree as specified and each rejection reason fires on its trigger (e.g. moving a node under its own descendant returns `Rejected{WouldCreateCycle}` and leaves the tree unchanged).
- **Est:** M

### M3-T03 ‚Äî Server-authoritative edit op-log with arrival-order serialization
- **Satisfies:** BUILD-2
- **Depends on:** M3-T02
- **Intent:** The server is the sole authority for a Design's edits: it assigns each accepted *edit op* a global sequence number in arrival order, appends to a per-Design *edit op-log*, and applies it to the canonical tree (Constitution Art. 1, no CRDT).
- **Touches:** `EditOpLog` (per-Design append-only list of sequenced ops), `SequencedEditOp{seq, sourcePlayerId, op}`, `DesignEditService.Submit(designId, playerId, EditOp) -> SubmitResult{seq | rejection}`, `DesignRegistry` (live Designs in memory keyed by design id).
- **Steps:**
  1. On `Submit`, take a per-Design lock/single-threaded queue so ops are totally ordered by arrival; assign the next monotonic `seq`.
  2. Resolve any client-proposed `newNodeId` for `AddPartOp` server-side: the server (not the client) assigns the authoritative stable node id and returns it in the result, so two concurrent adds can never collide on ids.
  3. Call `PartTreeMutator.Apply`; on `Applied`, append `SequencedEditOp` to the log and advance the Design's applied seq; on `Rejected`, do not append and return the reason.
  4. Keep the canonical `PartTree` and `EditOpLog` consistent: the log is the ordered record, the tree is the fold of the log; expose `seq` of last applied op.
- **Acceptance:** Feeding two interleaved op streams from two simulated players through one `DesignEditService` yields a single op-log whose sequence numbers are contiguous and whose folded tree equals replaying the log in seq order; client-proposed colliding add ids are both honored with distinct server-assigned node ids.
- **Est:** L

### M3-T04 ‚Äî Edit op wire protocol and broadcast to all editors
- **Satisfies:** BUILD-2
- **Depends on:** M3-T03
- **Intent:** Carry *edit ops* client‚Üíserver and broadcast each accepted `SequencedEditOp` to every player editing that Design, so all editors converge on the same tree in the same order.
- **Touches:** `EditOpMessage` (client‚Üíserver submit request, with client-local temp id for add), `EditOpAck` (server‚Üísubmitter: temp id ‚Üí server seq + assigned node id, or rejection), `EditOpBroadcast` (server‚Üíall editors: `SequencedEditOp`), `DesignEditorSession` (set of player ids subscribed to a design id), `DesignReplicationChannel`.
- **Steps:**
  1. Add join/leave-design messages that register a player into a Design's `DesignEditorSession` and reply with a full tree snapshot + current `seq` (resync baseline).
  2. On submit, the server returns an `EditOpAck` to the submitter (mapping their client temp id to the authoritative seq + node id) and emits an `EditOpBroadcast` to all session members including the submitter.
  3. Clients apply incoming broadcasts in `seq` order, buffering any out-of-order arrival until the gap fills; a client detecting a missing seq requests a fresh tree snapshot resync.
  4. Reuse the existing replication transport but on a Design-scoped logical channel ‚Äî this stays in the Design system and never enters the vessel snapshot/interpolation path (Art. 7).
- **Acceptance:** Two clients join one Design; each issues a concurrent add/remove/move; both clients and the server show byte-identical trees after broadcasts settle, and the demo "two players edit one rocket Design at once; edits serialize cleanly" holds.
- **Est:** L

### M3-T05 ‚Äî Persist Design and op-log; restore on restart
- **Satisfies:** BUILD-1, BUILD-2
- **Depends on:** M3-T03
- **Intent:** A Design and its *edit op-log* are durable state, stored in the single authoritative Postgres store (ADR-0007) via write-through, and restored exactly on server restart (PERSIST-style durability for the construction path).
- **Touches:** `design` table (design id, name, current seq, serialized `PartTree` as JSONB), `design_op` table (design id, seq, source player, op JSONB) or folded-tree-only with op-log capped, `DesignStore` (write-through repository), `DesignRegistry` load-on-demand.
- **Steps:**
  1. Define schema: one `design` row per Design holding the folded tree as JSONB plus last `seq`; optionally an append-only `design_op` table for audit/replay.
  2. Write through on each accepted op (or batched per submit) so the persisted tree + seq never lag the in-memory canonical tree past a single op.
  3. On Design load (first editor joins), hydrate the in-memory `PartTree` and seq from Postgres; on cold start the `DesignRegistry` is empty and lazy-loads.
  4. Keep the write transactional so a crash mid-write leaves a consistent (tree, seq) pair, never a torn state.
- **Acceptance:** After a series of edits, killing and restarting the server then re-joining the Design shows the identical tree and the same next `seq`; no op is lost or double-applied.
- **Est:** M

## Slice 3.2 ‚Äî Subtree locks

### M3-T06 ‚Äî Subtree lock manager: claim and release
- **Satisfies:** BUILD-3
- **Depends on:** M3-T03
- **Intent:** Let a player place an advisory *subtree lock* on a subassembly node and release it, with the server as the single authority over who holds which lock.
- **Touches:** `SubtreeLockManager` (per-Design map: locked root node id ‚Üí holder player id), `ClaimLockRequest{designId, nodeId}`, `ReleaseLockRequest{designId, nodeId}`, `LockResult` (Granted | Denied{heldBy}), `DesignEditService` integration.
- **Steps:**
  1. On claim, grant a lock on a node only if neither that node, any ancestor, nor any descendant is already locked by another player (nested/overlapping locks across different holders are denied); record holder.
  2. On release, allow only the holder (or server cleanup) to release; clear the entry.
  3. Auto-release a player's locks when they leave the `DesignEditorSession` or disconnect, so no seat/lock is reserved for an absent player.
  4. Serialize claim/release through the same per-Design ordering as edit ops so lock state and tree state share one consistent timeline.
- **Acceptance:** Player A claims a subtree and the manager reports it held by A; B's claim on that node or any node within it returns `Denied{heldBy=A}`; on A's disconnect the lock auto-releases and B's re-claim is `Granted`.
- **Est:** M

### M3-T07 ‚Äî Enforce subtree locks against foreign edit ops
- **Satisfies:** BUILD-3
- **Depends on:** M3-T02, M3-T06
- **Intent:** Reject any *edit op* from a player other than the lock holder when the op targets a node inside a locked subtree, while the holder's own ops still apply.
- **Touches:** `DesignEditService.Submit` (lock pre-check before mutation), `EditOpResult` rejection reason `SubtreeLocked{heldBy}`, `PartTree` ancestor-walk to find covering lock.
- **Steps:**
  1. Before applying any op, resolve the op's target node(s) ‚Äî `Add` targets its parent, `Remove`/`Move` target the node (and `Move` also its destination parent) ‚Äî and walk ancestors to find any covering lock.
  2. If a covering lock is held by a different player, reject with `SubtreeLocked{heldBy}` and do not append to the op-log.
  3. If the covering lock is held by the submitting player (or no lock covers the targets), proceed to normal apply.
  4. Broadcast lock state changes to all editors so clients can render which subtrees are claimed (advisory UI hint) and pre-empt rejected ops.
- **Acceptance:** While A holds a lock on the upper stage, B's move/remove op on a node in that subtree is rejected with `SubtreeLocked{heldBy=A}` and the tree is unchanged; A's own op on the same subtree applies; after A releases, B's op applies. Matches demo "one locks the upper stage; edits serialize cleanly."
- **Est:** M

## Slice 3.3 ‚Äî Launch

### M3-T08 ‚Äî Launch instantiates a Design into a Vessel
- **Satisfies:** BUILD-4
- **Depends on:** M3-T01
- **Intent:** *Launch* reads a finalized *Design* and instantiates a new *Vessel* in the world (on the pad) ‚Äî the one-way boundary from the design-time system into the flight/physics system, leaving the Design unchanged (Constitution Art. 7; Design and Vessel are distinct entities).
- **Touches:** `LaunchInstantiator` (consumes a `Design`, produces a `Vessel`), `LaunchRequest{designId, pad/spawn site}`, `VesselFactory` (flight-side; converts part tree ‚Üí *rigid vessel* with mass/resources, sets initial pose at the pad, creates empty *station* slots). The Design system hands off a read-only snapshot; it does not import flight types and flight does not import Design types ‚Äî they meet only in `LaunchInstantiator`.
- **Steps:**
  1. Take an immutable snapshot of the Design's `PartTree` at launch time so concurrent edits cannot mutate the in-flight instantiation.
  2. Translate the part tree into the flight-side Vessel representation: compute aggregate mass/dry-mass/resource capacities, derive the initial rigid-body composition (single rigid body split only at decouplers/stages), and place it at the requested pad pose with zero velocity.
  3. Create the Vessel's *station* slots (Pilot/Engineer/Navigator) unoccupied, and register the Vessel in the world so it appears to clients via the normal vessel path.
  4. Confirm the source Design is untouched (no op appended, seq unchanged) and persist the new Vessel through the world store.
- **Acceptance:** Launching a Design produces a new Vessel that appears in the world on the pad with the expected part count/mass and unoccupied station slots; the source Design's tree and op `seq` are unchanged; flight and construction assemblies link only through `LaunchInstantiator`. Matches demo "launch puts the craft on the pad as a Vessel."
- **Est:** L
