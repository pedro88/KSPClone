# M0 — Skeleton — code-ready tasks

> Expanded from roadmap.md slices 0.1–0.5. Each ticket cites the spec requirement(s) it satisfies. Terms in *italics* are in CONTEXT.md.

## Slice 0.1 — Fixed-step headless server + master clock

### M0-T01 — Stand up the engine-agnostic Sim core assembly and headless boot path
- **Satisfies:** NET-1
- **Depends on:** —
- **Intent:** Create the pure-C# *Sim core* library (no `UnityEngine` references) and a thin Unity host that loads it, so the same simulation logic compiles for both the headless server and clients per ADR-0006/0008.
- **Touches:** Sim core (engine-agnostic C#); `SimCore` asmdef, `ServerBootstrap` (Unity host MonoBehaviour), `SimWorld` (root aggregate holding clock + vessels).
- **Steps:**
  1. Create a Unity assembly definition `SimCore.asmdef` with `noEngineReferences = true`; place all simulation types (clock, propagator, warp FSM later) inside it so the compiler enforces engine-independence.
  2. Add a `SimWorld` root object that owns the (later) `MasterClock` and a `Dictionary<VesselId, Vessel>`; expose `Tick(double dtSeconds)` as the single advance entry point.
  3. Add a `ServerBootstrap` MonoBehaviour in the Unity assembly that instantiates one `SimWorld` and holds the authoritative reference (the only `SimWorld` on the server).
  4. Wire `ServerBootstrap.Awake()` to log `"[server] SimWorld constructed, authoritative=true"`.
- **Acceptance:** A unit test referencing `SimCore` compiles and constructs a `SimWorld` with zero `UnityEngine` dependency (assembly has no engine reference); building the player logs the authoritative-construction line exactly once.
- **Est:** M

### M0-T02 — Configure the Unity dedicated-server build target (`-batchmode -nographics`)
- **Satisfies:** NET-1
- **Depends on:** M0-T01
- **Intent:** Produce a headless Linux server build that runs with no renderer, so the *authoritative state* lives on a dedicated process per ADR-0001/0008 and Constitution Art. 1.
- **Touches:** Sim core boot; Unity build pipeline; `ServerBuilder` (editor build script), `ServerBootstrap`.
- **Steps:**
  1. Set the build to the Dedicated Server subtarget (or Linux headless player) via an editor `ServerBuilder` menu/CLI script using `BuildPipeline.BuildPlayer` with `StandaloneBuildSubtarget.Server`.
  2. In `ServerBootstrap`, assert `Application.isBatchMode` at startup; if a renderer/camera would be active, log a warning so a mis-launch is visible.
  3. Set `Application.targetFrameRate = -1` and disable VSync so the host loop is not throttled to a display refresh.
  4. Provide a launch command/script invoking the built binary with `-batchmode -nographics -logFile <path>`.
- **Acceptance:** Running the built binary with `-batchmode -nographics` starts the process, writes the log file, prints the authoritative-construction line, and exits cleanly on SIGTERM — with no graphics device created (no GL/Vulkan init lines in the log).
- **Est:** M

### M0-T03 — Implement the fixed 60 Hz accumulator scheduler decoupled from frame rate
- **Satisfies:** NET-5
- **Depends on:** M0-T01
- **Intent:** Drive `SimWorld.Tick` from a fixed-timestep accumulator so gameplay never depends on render framerate, per ADR-0006 and Constitution Art. 2.
- **Touches:** Sim core scheduler; `SimScheduler`, `SimWorld`.
- **Steps:**
  1. Implement `SimScheduler` with `const double FixedDt = 1.0 / 60.0` and a `double _accumulator`.
  2. Each host update, add real elapsed seconds to `_accumulator`; while `_accumulator >= FixedDt`, call `SimWorld.Tick(FixedDt)` and subtract `FixedDt`.
  3. Clamp the per-update added time (spiral-of-death guard, e.g. max 0.25 s = 15 catch-up ticks) and log a warning when the clamp fires.
  4. Maintain a monotonic `long TickCount`; expose it for tests and logging.
  5. Drive `SimScheduler.Advance(deltaTime)` from `ServerBootstrap.Update()` using `Time.unscaledDeltaTime`; keep the simulation source-of-time independent of `Time.timeScale`.
- **Acceptance:** A test feeding 1.0 s of accumulated real time in arbitrary irregular chunks (e.g. 0.003 s and 0.05 s mixed) produces exactly 60 `Tick` calls of `FixedDt` each (±0 ticks), and `TickCount` increases monotonically.
- **Est:** M

### M0-T04 — Prove tick stability under artificial render stalls
- **Satisfies:** NET-5
- **Depends on:** M0-T03
- **Intent:** Demonstrate that injected frame stalls (long render frames) do not change simulated tick cadence, only catch-up batching — closing T2's acceptance check.
- **Touches:** Sim core scheduler; `SimScheduler`, test harness `SchedulerStallTests`.
- **Steps:**
  1. Add a test that simulates a 250 ms stall by passing a single large `Advance(0.25)` then resuming normal `1/60` chunks.
  2. Assert the catch-up loop runs the correct number of ticks (clamped per M0-T03) and that average tick interval over a 10 s simulated window equals `FixedDt` within ±1 tick.
  3. Add an in-server diagnostic that logs measured ticks-per-second once per simulated second.
- **Acceptance:** Over a 10 s simulated run with two injected stalls, measured mean ticks/s is 60 ±1 and no tick is shorter or longer than `FixedDt`; the per-second tick-rate log stays at 60 except the explicit catch-up second.
- **Est:** S

### M0-T05 — Implement the single authoritative `MasterClock` advancing at baseline rate
- **Satisfies:** TIME-1, TIME-2
- **Depends on:** M0-T03
- **Intent:** Provide exactly one *master clock* holding *game-time* that advances 1:1 with real-time at *baseline rate* while no *warp* is active, including with zero players, per Constitution Art. 1/4.
- **Touches:** Sim core clock; `MasterClock` (singleton-per-`SimWorld`), `SimWorld.Tick`.
- **Steps:**
  1. Implement `MasterClock` with `double GameTimeSeconds` and `double Rate` (default `1.0` = baseline).
  2. In `SimWorld.Tick(dt)`, advance `GameTimeSeconds += dt * Rate` as the only writer of game-time; expose a read-only accessor.
  3. Construct exactly one `MasterClock` in `SimWorld`'s constructor; add a guard/test that no second instance is created.
  4. Keep advancing regardless of connected-player count (no gating on session state).
- **Acceptance:** A test advancing the scheduler for 60 s of simulated real-time with `Rate=1.0` and zero clients leaves `GameTimeSeconds` at 60.0 ±`FixedDt` (±1 tick); a test asserts only one `MasterClock` exists per `SimWorld`.
- **Est:** S

## Slice 0.2 — On-rails vessel + patched conics

### M0-T06 — Define the celestial body / SOI data model
- **Satisfies:** ORBIT-1
- **Depends on:** M0-T01
- **Intent:** Model the gravitational hierarchy so each vessel is under exactly one body's gravity (its current *SOI*), per *patched conics* / ADR-0004.
- **Touches:** Sim core orbital model; `CelestialBody` (id, `GravParameterMu`, `SoiRadius`, parent ref, orbit-around-parent), `BodyRegistry`.
- **Steps:**
  1. Implement immutable `CelestialBody` with `double GravParameterMu` (μ = GM), `double SoiRadius`, optional parent `CelestialBodyId`, and its own orbital elements about its parent.
  2. Implement `BodyRegistry` holding a small fixed system for M0 (one central star/planet + one moon, or planet + moon) and lookup by id.
  3. Provide `BodyRegistry.WorldPositionOf(bodyId, gameTime)` returning the body's position at a *game-time* (root body fixed at origin).
  4. Seed deterministic constants (μ, SOI radii, semi-major axes) in a config asset/JSON so values are reproducible across restarts.
- **Acceptance:** A test loads the registry, asserts the system has ≥2 bodies with a defined parent chain to the root, and that each body's `SoiRadius` and `GravParameterMu` are positive; `WorldPositionOf(root, t)` is constant at origin for all `t`.
- **Est:** M

### M0-T07 — Define the `Orbit` (Keplerian elements) and `Vessel` data structures
- **Satisfies:** ORBIT-1
- **Depends on:** M0-T06
- **Intent:** Represent a vessel's analytic conic around its current *SOI* body and the *vessel* record itself, so a vessel persists an orbit per ORBIT-1.
- **Touches:** Sim core orbital model; `Orbit` (a, e, i, Ω, ω, M0, epoch, parent body), `Vessel` (id, `Orbit`, *vessel clock*, on-rails flag), `SimWorld` vessel registry.
- **Steps:**
  1. Implement `Orbit` as the six classical elements — semi-major axis `a`, eccentricity `e`, inclination `i`, longitude of ascending node `Ω`, argument of periapsis `ω`, mean anomaly at epoch `M0` — plus `double EpochGameTime` and `CelestialBodyId ParentBody`.
  2. Implement `Vessel` holding a stable `VesselId`, an `Orbit`, a `VesselClockSeconds` (= master clock while on-rails), and `bool OnRails = true` (default per Constitution Art. 3).
  3. Add one seeded vessel to `SimWorld` on boot with a stable circular/elliptical orbit around a chosen body.
  4. Provide `Orbit.MeanMotion` = `sqrt(mu / a^3)` as a derived property for downstream propagation.
- **Acceptance:** A test constructs the seeded vessel, reads back all six orbital elements and parent body unchanged, confirms `OnRails == true` by default, and asserts `MeanMotion` matches `sqrt(mu/a^3)` for the seeded values within 1e-9.
- **Est:** M

### M0-T08 — Implement the closed-form Kepler propagator `position(game-time)`
- **Satisfies:** ORBIT-2
- **Depends on:** M0-T07
- **Intent:** Compute an *on-rails vessel*'s state vector at any future *game-time* in closed form with no step integration, the load-bearing property from ADR-0004 / Constitution Art. 3.
- **Touches:** Sim core propagator; `KeplerPropagator`, `Orbit`.
- **Steps:**
  1. Compute mean anomaly at time `t`: `M = M0 + MeanMotion * (t - EpochGameTime)`, wrapped to `[0, 2π)`.
  2. Solve Kepler's equation `M = E - e*sin(E)` for eccentric anomaly `E` via Newton-Raphson (`E_{n+1} = E_n - (E_n - e sin E_n - M)/(1 - e cos E_n)`), seed `E0 = M`, iterate to `|f| < 1e-12` or 50 iters; handle `e ≥ 1` (hyperbolic) with the hyperbolic Kepler form `M = e*sinh(F) - F`.
  3. Convert to true anomaly `ν` and radius `r = a(1 - e cos E)`; build the perifocal position/velocity, then rotate to the parent body's inertial frame via the 3-1-3 rotation (Ω, i, ω).
  4. Offset by the parent body's world position (M0-T06) to yield world-frame `position(t)` and `velocity(t)`.
  5. Expose `KeplerPropagator.StateAt(Orbit, double gameTime) -> (Vector3d pos, Vector3d vel)`; use double precision throughout.
- **Acceptance:** For a seeded eccentric orbit, `StateAt(t + Δ)` for several Δ (including > one period and a multi-day jump) matches a reference RK4 numeric integration of the two-body problem to within 1 m position / 1e-3 m/s velocity, and the call performs **no** time-stepping (single evaluation, asserted by call-count/instrumentation).
- **Est:** L

### M0-T09 — Sync on-rails vessels to the master clock each tick (no integration)
- **Satisfies:** ORBIT-2, SUSP-1
- **Depends on:** M0-T08, M0-T05
- **Intent:** Keep every *on-rails vessel*'s evaluated state synced to the *master clock* by closed-form evaluation, so the *living universe* stays current including while empty (SUSP-1).
- **Touches:** Sim core; `SimWorld.Tick`, `KeplerPropagator`, `Vessel`.
- **Steps:**
  1. In `SimWorld.Tick`, for each `Vessel` with `OnRails == true`, set `VesselClockSeconds = MasterClock.GameTimeSeconds` (rails stay synced — no lag).
  2. Cache the latest evaluated `StateAt(orbit, GameTime)` on the vessel for consumers (snapshot emitter, POI scan) — recompute lazily, do not integrate.
  3. Ensure this path runs with zero connected players (driven purely by the scheduler).
- **Acceptance:** Advancing the clock 1 game-day with zero clients, the seeded vessel's cached state equals `StateAt(orbit, finalGameTime)` exactly (same code path), and its `VesselClockSeconds == MasterClock.GameTimeSeconds`; CPU profile shows no per-step integration calls.
- **Est:** S

### M0-T10 — Detect and register SOI-crossing POIs
- **Satisfies:** ORBIT-3
- **Depends on:** M0-T08
- **Intent:** Predict when an *on-rails vessel* will cross an *SOI* boundary and record that *game-time* as a *POI*, so warp auto-limit (later) can stop on it (ORBIT-3).
- **Touches:** Sim core orbital model; `SoiScanner`, `PoiRegistry`, `Poi` (type, gameTime, vesselId), `Orbit`.
- **Steps:**
  1. For each on-rails vessel, sample distance-to-parent vs. parent `SoiRadius` and distance-to-each-sibling/child-body vs. that body's `SoiRadius` along the orbit, looking forward from current game-time.
  2. Locate the sign-change of `f(t) = r_to_target(t) - SoiRadius_target` by coarse stepping in mean-anomaly/time, then refine the crossing time with bisection/Newton to ≤ 1 s precision (closed-form `StateAt`, no integration).
  3. Create a `Poi { Type = SoiCrossing, GameTime, VesselId, FromBody, ToBody }` and insert it into `PoiRegistry`, keyed/sorted by `GameTime`.
  4. Re-scan and update a vessel's POI when its orbit changes (e.g. after a re-parent in M0-T11).
- **Acceptance:** For a seeded transfer orbit that enters a moon's SOI, `SoiScanner` registers a `SoiCrossing` POI whose predicted `GameTime` matches the crossing found by fine numeric reference to within 1 s; the POI appears in `PoiRegistry.Earliest()`.
- **Est:** L

### M0-T11 — Re-parent the orbit on SOI crossing
- **Satisfies:** ORBIT-3
- **Depends on:** M0-T10
- **Intent:** When a vessel actually reaches a registered SOI crossing, re-express its *orbit* about the new body so patched conics stays consistent (ORBIT-3).
- **Touches:** Sim core; `SoiTransition`, `KeplerPropagator`, `Orbit`, `PoiRegistry`.
- **Steps:**
  1. When `MasterClock.GameTime >= poi.GameTime` for an `SoiCrossing`, take the vessel's world state from `StateAt` at exactly `poi.GameTime`.
  2. Subtract the new parent body's world position/velocity at that time to get the relative state in the new body's frame.
  3. Convert the relative state vector back to classical elements (`StateVectorToOrbit`) about the new body, with `EpochGameTime = poi.GameTime`; set `Orbit.ParentBody = poi.ToBody`.
  4. Remove the consumed POI and trigger a re-scan (M0-T10) for the new orbit; emit a write-through event hook (consumed in M0-T13).
- **Acceptance:** A test runs a vessel up to a registered SOI crossing; at the crossing the vessel's `ParentBody` changes to the target body and its position/velocity are **continuous** across the switch (world-frame state before vs. after differ by < 1 m / 1e-3 m/s); a new POI is rescanned for the new orbit.
- **Est:** L

## Slice 0.3 — Client connect + observe

### M0-T12 — Establish the client↔server transport and connection lifecycle
- **Satisfies:** NET-1
- **Depends on:** M0-T02
- **Intent:** Open a reliable channel from a client to the dedicated server with connect/disconnect handling and a dev-stub identity (per plan P-3), keeping the server the sole authority (NET-1).
- **Touches:** Replication layer; `ServerNetHost`, `ClientNetPeer`, `ConnectionRegistry`, dev-stub `PlayerSession`.
- **Steps:**
  1. Choose the custom transport (e.g. raw UDP/ENet-style or LiteNetLib-class library — not Unity high-level netcode, per ADR-0008) and define a `ServerNetHost` that listens and a `ClientNetPeer` that connects.
  2. On connect, register a `PlayerSession` with a generated `PlayerId` (dev stub: accept any peer, no auth — P-3) in a `ConnectionRegistry`; log `"[server] player <id> connected (n=<count>)"`.
  3. On disconnect/timeout, remove the session and log the new count; expose `ConnectionRegistry.ConnectedCount` (needed by the warp vote in Slice 0.4).
  4. Keep all simulation writes server-side; the client peer is receive/observe-only in M0.
- **Acceptance:** A loopback integration test connects one `ClientNetPeer`, asserts `ConnectedCount == 1` and a connect log line, then disconnects and asserts `ConnectedCount == 0`; the client holds no authoritative `SimWorld`.
- **Est:** M

### M0-T13 — Send the world handshake on connect (vessel list + current game-time)
- **Satisfies:** NET-1
- **Depends on:** M0-T12, M0-T07, M0-T05
- **Intent:** On connect, send the client a snapshot of the world so it can list the vessel and display current *game-time*, closing T7 (NET-1).
- **Touches:** Replication layer; `WorldHandshakeMessage`, `ServerNetHost`, `ClientWorldModel`.
- **Steps:**
  1. Define a `WorldHandshakeMessage` carrying `MasterClock.GameTimeSeconds`, the body registry summary, and for each vessel its `VesselId` + current `Orbit` elements (or current state vector).
  2. Serialize and send it to a peer immediately after `PlayerSession` registration.
  3. On the client, populate a read-only `ClientWorldModel` from the handshake (vessel list, clock).
  4. Log on the client `"[client] handshake: <vesselCount> vessels, gameTime=<t>"`.
- **Acceptance:** A loopback test connects a client and asserts its `ClientWorldModel` lists exactly the seeded vessel(s) and a `GameTimeSeconds` within one snapshot interval of the server's clock at connect time.
- **Est:** M

### M0-T14 — Emit authoritative snapshots at 20–30 Hz
- **Satisfies:** NET-5, NET-4
- **Depends on:** M0-T13, M0-T09
- **Intent:** Stream periodic *authoritative* vessel state to connected clients at the spec rate, decoupled from the 60 Hz sim tick, per NET-5 / ADR-0006.
- **Touches:** Replication layer; `SnapshotEmitter`, `VesselSnapshot` (vesselId, pos, vel, gameTime, seq), `ServerNetHost`.
- **Steps:**
  1. Add a `SnapshotEmitter` driven by the sim tick with its own accumulator targeting a configurable 20–30 Hz (default 25 Hz, i.e. every ~2.4 ticks).
  2. Each emission, build a `VesselSnapshot` per vessel from the cached on-rails state (M0-T09), stamped with `MasterClock.GameTimeSeconds` and a monotonically increasing `Seq`.
  3. Send snapshots to all sessions in `ConnectionRegistry` over the transport (unreliable/sequenced channel).
  4. Log/measure actual emission rate once per second for verification.
- **Acceptance:** With a connected client, measured snapshot rate at the client is 20–30 Hz over a 10 s window, snapshots carry increasing `Seq` and `gameTime`, and emission cadence is independent of the 60 Hz tick (asserted by differing counters).
- **Est:** M

### M0-T15 — Client-side interpolation of non-controlled vessels
- **Satisfies:** NET-4
- **Depends on:** M0-T14
- **Intent:** Render the observed vessel smoothly by *interpolating* buffered *authoritative* snapshots with a small delay (no prediction in M0), per NET-4.
- **Touches:** Client presentation / Replication layer; `SnapshotBuffer`, `VesselInterpolator`, `ClientWorldModel`.
- **Steps:**
  1. Buffer incoming `VesselSnapshot`s per vessel in a time-ordered `SnapshotBuffer`.
  2. Maintain a render clock running `interpolationDelay` (e.g. 100 ms, ≈ 2–3 snapshot intervals) behind the latest received `gameTime`.
  3. Each render frame, find the two snapshots bracketing the render time and linearly interpolate position (and slerp/lerp orientation if present); extrapolate briefly if the buffer underruns, then hold.
  4. Drive the vessel's rendered transform from the interpolated state (render-frame-rate, fully decoupled from the 60 Hz sim — Constitution Art. 2).
- **Acceptance:** With snapshots arriving at 25 Hz, the rendered vessel position at 60 fps is smooth (no per-snapshot stairstep > a defined epsilon) and lags the latest authoritative snapshot by `interpolationDelay` ±1 frame; a test feeding two snapshots asserts the midpoint render equals their linear interpolation.
- **Est:** M

## Slice 0.4 — Warp vote + auto-limit

### M0-T16 — Implement the warp request + unanimous vote state machine
- **Satisfies:** TIME-3
- **Depends on:** M0-T12, M0-T05
- **Intent:** Gate any *warp* behind a unanimous *warp vote* — it begins only after every connected player approves — per TIME-3 and CONTEXT *Warp vote*.
- **Touches:** Sim core warp FSM; `WarpStateMachine` (states: Idle, Voting, Active), `WarpVote`, `RequestWarpMessage` / `VoteMessage`, `ConnectionRegistry`.
- **Steps:**
  1. Define `WarpStateMachine` with states `Idle → Voting → Active → Idle`; a `RequestWarpMessage(rate)` from a client transitions `Idle → Voting` and opens a `WarpVote` over the current connected set.
  2. Track per-player approvals; a non-approval is "not yet," not a permanent veto (vote stays open). Transition `Voting → Active` only when approvals == `ConnectionRegistry.ConnectedCount`.
  3. Broadcast vote state (pending voters) to all clients each change so UI can show progress.
  4. Apply the warp by setting `MasterClock.Rate` to the requested multiplier on entering `Active` (single writer of `Rate`).
  5. Provide a cancel/return-to-Idle path (requester cancels or auto-limit ends it — latter in M0-T19).
- **Acceptance:** An integration test with 3 connected clients: warp does **not** start with 2/3 approvals; on the 3rd approval the state goes `Active` and `MasterClock.Rate` becomes the requested multiplier; a withheld approval keeps it in `Voting` indefinitely.
- **Est:** L

### M0-T17 — Plumb the two warp kinds (physics vs on-rails) with rate selection
- **Satisfies:** TIME-7
- **Depends on:** M0-T16
- **Intent:** Distinguish *physics warp* (low multiplier, physics still stepping) from *on-rails warp* (high multiplier, analytic) — both gated by the TIME-3 vote — per TIME-7.
- **Touches:** Sim core warp FSM; `WarpKind` enum, `WarpStateMachine`, `MasterClock`.
- **Steps:**
  1. Add `enum WarpKind { Physics, OnRails }`; classify the requested multiplier (e.g. ≤ x4 ⇒ Physics, ≥ x1000 ⇒ OnRails) or carry the kind explicitly in `RequestWarpMessage`.
  2. For `OnRails` warp, require all in-scope vessels to be *warp-safe* (in M0 all are on-rails by default) before allowing `Active`; refuse otherwise with a logged reason.
  3. Set `MasterClock.Rate` to the multiplier; since M0 has only on-rails vessels, advancing the clock fast simply re-evaluates closed-form `position(t)` each tick — no integration (Constitution Art. 3).
  4. Tag the active warp with its `WarpKind` for clients/telemetry.
- **Acceptance:** A test runs an on-rails warp at x1000 for a span of game-time: `MasterClock.GameTime` advances ~1000× wall-time, the seeded vessel's state at the end equals `StateAt(orbit, finalGameTime)` (still on its analytic orbit, no drift), and the active `WarpKind == OnRails`.
- **Est:** M

### M0-T18 — Scan all vessels for the earliest global POI
- **Satisfies:** TIME-4
- **Depends on:** M0-T10, M0-T16
- **Intent:** Compute the soonest *POI* across **all** vessels in the world, the value the *auto-limit* clamps to (TIME-4).
- **Touches:** Sim core; `PoiRegistry.Earliest`, `WarpStateMachine`, `SoiScanner`.
- **Steps:**
  1. Ensure every vessel's upcoming POIs (SOI crossings from M0-T10; maneuver nodes / atmospheric interfaces are later milestones) are present in `PoiRegistry`.
  2. Implement `PoiRegistry.EarliestAfter(double gameTime)` returning the minimum-`GameTime` POI across all vessels strictly after the given time.
  3. Refresh the earliest-POI query when a warp request is accepted (and when orbits re-parent mid-warp).
- **Acceptance:** With two seeded vessels having SOI crossings at different game-times, `EarliestAfter(now)` returns the earlier of the two; adding a sooner POI for a third vessel changes the result accordingly.
- **Est:** S

### M0-T19 — Clamp warp end-time to the earliest global POI (auto-limit)
- **Satisfies:** TIME-4
- **Depends on:** M0-T18, M0-T17
- **Intent:** Even after a vote passes, clamp the warp's actual end *game-time* to the soonest *POI* so no vessel silently skips its crossing/burn (TIME-4, *auto-limit*).
- **Touches:** Sim core warp FSM; `WarpStateMachine`, `MasterClock`, `PoiRegistry`.
- **Steps:**
  1. On entering `Active`, compute `endTime = PoiRegistry.EarliestAfter(GameTime)` (if any) as the auto-limit target; store it on the warp.
  2. Each tick while `Active`, after advancing the clock, if `MasterClock.GameTime >= endTime`, set `GameTime = endTime` exactly, reset `Rate = 1.0`, and transition `Active → Idle` (halt to baseline).
  3. Guard the final tick so the clock lands **at** the POI, never past it (clamp the last `dt`).
  4. Emit a warp-commit event at the clamped endpoint (consumed by persistence in M0-T22).
- **Acceptance:** A test starts an on-rails warp toward a vessel with an SOI crossing at `T_poi`; the warp halts with `MasterClock.GameTime == T_poi` exactly (never `> T_poi`), `Rate` returns to 1.0, and the state machine is back in `Idle`.
- **Est:** M

### M0-T20 — Maintain vote membership across connect/disconnect during warp
- **Satisfies:** TIME-5, TIME-6
- **Depends on:** M0-T16
- **Intent:** A player disconnecting mid-warp stops being a voter and warp continues; a player connecting mid-warp never consented, so warp halts to baseline and they join the next vote — per TIME-5/TIME-6.
- **Touches:** Sim core warp FSM; `WarpStateMachine`, `ConnectionRegistry`, `WarpVote`.
- **Steps:**
  1. Subscribe `WarpStateMachine` to `ConnectionRegistry` connect/disconnect events.
  2. On disconnect while `Voting`: remove the player from the required set and re-evaluate unanimity (may now pass). On disconnect while `Active`: drop them as a voter; warp continues unchanged (TIME-5).
  3. On connect while `Active`: immediately transition `Active → Idle`, set `Rate = 1.0` (halt to baseline), and include the new player in the next vote (TIME-6).
  4. On connect while `Voting`: add them to the required set (vote now needs their approval too).
  5. Log each membership change with reason for verification.
- **Acceptance:** Integration test (3 clients, warp active): disconnecting one keeps `Rate` unchanged and warp `Active` (TIME-5); a fresh client connecting forces `Rate → 1.0` and state `Idle` within one tick (TIME-6), and that client is a required voter in the next `RequestWarp`.
- **Est:** M

## Slice 0.5 — Persistence

### M0-T21 — Define the Postgres schema and write-through repository layer
- **Satisfies:** PERSIST-1
- **Depends on:** M0-T07, M0-T06
- **Intent:** Create the single authoritative Postgres store for all durable state (program, vessel, clock) with an in-memory-hot + write-through repository, per ADR-0007 / PERSIST-1 / Constitution Art. 8.
- **Touches:** Persistence layer; migrations `001_init.sql`, `WorldRepository`, `program` / `vessel` / `world_clock` tables, Npgsql data access.
- **Steps:**
  1. Write migration `001_init.sql` creating: `program(id, mode, science, funds, created_at)`; `vessel(id, parent_body, sma, ecc, inc, raan, argp, mean_anom_epoch, epoch_game_time, vessel_clock, on_rails, snapshot JSONB NULL)`; `world_clock(id PRIMARY KEY CHECK(id=0), game_time DOUBLE PRECISION, warp_rate)` (single-row).
  2. Store orbital elements as columns (cheap rows per ADR-0007) and reserve the `snapshot JSONB` column for suspended active vessels (M1).
