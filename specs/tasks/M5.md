# M5 — Progression (Science mode) — code-ready tasks

> Expanded from roadmap.md slices 5.1–5.2. Each ticket cites the spec requirement(s) it satisfies. Terms in *italics* are in CONTEXT.md.

> **Career deferred.** Career (*funds* treasury + contracts, PROG-4 Career arm) is intentionally out of the first cut. These tickets build the *space program* data model and APIs so Career layers on without rework: the schema carries a `funds` column and a `game_mode` that admits `career`, but no contract/funds gameplay is implemented here. Do not write Career tickets.

> **Constitutional anchors.** Art. 8 — one shared *space program* (one *tech tree*, one *science* pool, one *funds* treasury), durable in a single Postgres store, transactional with world state (ADR-0007). Art. 1 — server is the sole authority; clients never mutate the pool directly. Art. 9 — every ticket cites a REQ-ID. Art. 10 — server → wire → client per slice.

## Slice 5.1 — Shared program + science + tech tree

### M5-T01 — Define the shared program Postgres schema (program, tech tree, unlock state)
- **Satisfies:** PROG-1, PERSIST-1
- **Depends on:** — (extends the `program` table introduced in M0 / roadmap T13)
- **Intent:** Lay down the single-row-per-world *space program* and the static *tech tree* graph plus its per-world unlock state, in the one Postgres store, with columns reserved for Career so it layers on without a migration rewrite.
- **Touches:** Postgres tables `program`, `tech_node`, `tech_node_unlock`; migration script; C# persistence types `ProgramRow`, `TechNodeRow`, `TechNodeUnlockRow` in the Persistence layer.
- **Steps:**
  1. Define `program` as exactly one row per world: `program_id PK`, `world_id UNIQUE` (enforces one program per world — Art. 8), `game_mode` enum/text constrained to `sandbox|science|career`, `science_pool NUMERIC NOT NULL DEFAULT 0`, `funds NUMERIC NOT NULL DEFAULT 0` (reserved for Career, unused now), `updated_at`.
  2. Define `tech_node` as the static graph: `tech_node_id PK`, `display_name`, `science_cost NUMERIC NOT NULL`, `prerequisites tech_node_id[]` (or a `tech_node_edge` join table), `unlocks_part_ids text[]`. This is content/config, not per-world state.
  3. Define `tech_node_unlock` as per-program unlock state: `(program_id FK, tech_node_id FK)` composite PK, `unlocked_at`, `unlocked_by_player_id` (nullable, audit only). A row's presence = unlocked for the whole program.
  4. Add the migration; seed a minimal tech tree (root node + 2–3 dependent nodes referencing real part ids) for tests.
  5. Add a `program` row uniqueness constraint check and a FK from `tech_node_unlock.program_id` to `program`.
- **Acceptance:** Migration applies on a clean DB; inserting a second `program` row for the same `world_id` is rejected by the unique constraint; seeded tech tree loads with prerequisite edges intact; a `funds` column exists and defaults to 0 with no code path writing it.
- **Est:** M

### M5-T02 — Load program + tech state into the hot in-memory model on server start
- **Satisfies:** PROG-1, PERSIST-1, PERSIST-3
- **Depends on:** M5-T01
- **Intent:** Give the *sim core* an authoritative in-memory view of the shared program (science pool, mode, unlocked node set) hydrated from Postgres at boot, matching ADR-0007's hot-copy + write-through model.
- **Touches:** C# `ProgramStore` (in-memory authoritative aggregate: pool, mode, unlocked set), `SciencePool`, `TechTree` (static graph loaded from `tech_node`), `ProgramRepository` (Postgres read/write) in Persistence; server bootstrap.
- **Steps:**
  1. Implement `ProgramRepository.Load(worldId)` → reads the `program` row, the full `tech_node` graph, and all `tech_node_unlock` rows for that program.
  2. Build `TechTree` from `tech_node` rows: nodes, prerequisite edges, costs, part-id payloads; expose `IsUnlockable(nodeId, unlockedSet)` (all prereqs unlocked) and `Cost(nodeId)`.
  3. Build `ProgramStore` holding `SciencePool` (current value), `GameMode`, and the unlocked-node set; expose read accessors only (mutations come in M5-T03/T04).
  4. Wire load into server bootstrap so the program is in memory before the first tick (Art. 1 — authority owns it).
- **Acceptance:** On a fresh server start against a DB seeded by M5-T01, `ProgramStore` reports the persisted `science_pool`, `game_mode`, and the exact set of unlocked nodes; restarting the server reproduces the identical in-memory state (PERSIST-3).
- **Est:** M

### M5-T03 — Replicate the program read-model to clients (snapshot + join handshake)
- **Satisfies:** PROG-1, NET-4
- **Depends on:** M5-T02
- **Intent:** Make the one shared pool, mode, and unlocked-node set visible to every connected client via the existing authoritative snapshot/handshake path, so all four players observe identical progression.
- **Touches:** Replication layer — extend world handshake (roadmap T7) and snapshot emitter (roadmap T8) with a `ProgramState` block (`sciencePool`, `gameMode`, `unlockedNodeIds`); client-side `ProgramReadModel`; wire DTO `ProgramStateDto`.
- **Steps:**
  1. Add `ProgramStateDto` to the wire schema: science pool value, game mode, unlocked node id list (send full list on handshake; on snapshots send the pool value + a delta/version of the unlocked set).
  2. Emit `ProgramState` in the join handshake so a connecting client gets the full current program immediately.
  3. Include the science-pool value (and unlock-set version) in the periodic snapshot so changes propagate within one snapshot interval.
  4. Implement client `ProgramReadModel` that stores the latest received program state; clients never mutate it locally (Art. 1).
- **Acceptance:** Two clients connect to the same world and both display the same `sciencePool`, `gameMode`, and unlocked-node set as authoritative Postgres; a fresh client joining mid-session receives the complete unlocked set in its handshake.
- **Est:** M

### M5-T04 — Earn science: award command adds to the shared pool, write-through
- **Satisfies:** PROG-2, PERSIST-2
- **Depends on:** M5-T02, M5-T03
- **Intent:** Let an in-world *science* award (from an experiment/exploration event) atomically increase the single shared *science* pool on the server and persist it, so any player's contribution benefits the whole *space program*.
- **Touches:** C# `ScienceLedger` / `ProgramStore.AwardScience(amount, sourceRef)`; `ProgramRepository.PersistSciencePool` (write-through); optional `science_award` audit table `(program_id, amount, source_ref, awarded_at)`; server command handler for an award event.
- **Steps:**
  1. Define an authoritative `AwardScience(amount, sourceRef)` API on `ProgramStore` that increments `SciencePool` in memory (server-only; reject any client-originated direct pool write — Art. 1).
  2. In the same transaction, write the new `program.science_pool` through to Postgres and append a `science_award` audit row (ties "science awarded" to its source per ADR-0007 atomic-consistency rationale; PERSIST-2).
  3. Bump the program-state version so the next snapshot carries the new pool value to clients.
  4. Reject negative or non-finite award amounts.
- **Acceptance:** Client A triggers a science award; the `program.science_pool` row increases by exactly that amount and a `science_award` audit row is written in the same transaction; Client B sees the updated pool within one snapshot interval.
- **Est:** M

### M5-T05 — Spend science: unlock a tech node for all players, transactional
- **Satisfies:** PROG-3, PERSIST-2
- **Depends on:** M5-T01, M5-T02, M5-T04
- **Intent:** Let the program spend pooled *science* to unlock a *tech node*, atomically debiting the pool and inserting the unlock so the node (and its parts) becomes available to every player at once.
- **Touches:** C# `ProgramStore.UnlockNode(nodeId, requestingPlayerId)`, `TechTree.IsUnlockable`; `ProgramRepository.PersistUnlock` (single transaction: debit pool + insert `tech_node_unlock`); server command handler for unlock requests.
- **Steps:**
  1. Validate the unlock request: node exists, not already unlocked, all prerequisites unlocked (`TechTree.IsUnlockable`), and `SciencePool >= Cost(nodeId)`.
  2. In one Postgres transaction: subtract `science_cost` from `program.science_pool` and insert the `tech_node_unlock` row (`unlocked_by_player_id` = requester, audit only). Roll back on any failure so the pool is never debited without an unlock (Art. 8, ADR-0007 atomicity).
  3. Apply the same mutation to the in-memory `ProgramStore` (pool down, node added to unlocked set) and bump the program-state version.
  4. Reject the command (no state change) if any validation fails, returning a typed refusal to the requester.
- **Acceptance:** With sufficient pool, Client A unlocks a node; the `tech_node_unlock` row is inserted and the pool is debited in the same transaction; Client B sees the node unlocked and the lowered pool within one snapshot. An unlock requested with insufficient science or unmet prerequisites is refused and leaves both pool and unlock set unchanged.
- **Est:** M

### M5-T06 — Restore program state across restart (round-trip durability check)
- **Satisfies:** PERSIST-3, PROG-1
- **Depends on:** M5-T04, M5-T05
- **Intent:** Prove the shared program survives a server restart exactly — pool, mode, and unlocked set restore identically — closing the PERSIST-3 loop for progression.
- **Touches:** No new schema; integration test harness exercising `ProgramRepository.Load` after a kill/restart; bootstrap path from M5-T02.
- **Steps:**
  1. In a test world: award science (M5-T04) and unlock at least one node (M5-T05).
  2. Kill and restart the server process.
  3. Reload via `ProgramRepository.Load` and compare `science_pool`, `game_mode`, and the unlocked-node set against the pre-restart values.
  4. Connect a client and confirm the handshake reports the restored state.
- **Acceptance:** After kill + restart, science pool, game mode, and unlocked-node set are byte-for-byte identical to pre-restart; a newly connected client's handshake reflects the restored program.
- **Est:** S

## Slice 5.2 — Modes

### M5-T07 — Game-mode flag drives unlock gating (Sandbox pre-unlocks, Science gates)
- **Satisfies:** PROG-4
- **Depends on:** M5-T02, M5-T05
- **Intent:** Make the `game_mode` flag govern part/tech availability: Sandbox treats every node as unlocked regardless of *science*; Science enforces the cost+prerequisite gate from M5-T05; Career is admitted by the enum but behaves as Science for now (funds/contracts deferred).
- **Touches:** C# `GameMode` enum (`Sandbox|Science|Career`), `ProgramStore.IsPartAvailable(partId)` / `IsNodeUnlocked(nodeId)`; the unlock-request handler from M5-T05; world-creation config that sets `program.game_mode`.
- **Steps:**
  1. Define `GameMode` and read it from `program.game_mode` at load (M5-T02).
  2. Implement availability resolution: in `Sandbox`, `IsNodeUnlocked` / `IsPartAvailable` return true for all nodes/parts (a pure flag — no rows inserted, pool untouched); in `Science` and (for now) `Career`, defer to the actual `tech_node_unlock` set.
  3. Make the unlock-request handler a no-op success in `Sandbox` (nothing to spend) and the gated transaction (M5-T05) in `Science`/`Career`.
  4. Surface `gameMode` and effective availability through the program read-model (M5-T03) so clients gate the parts catalog/VAB by the same rule.
- **Acceptance:** A world created with `game_mode=sandbox` reports every node available and every part usable with a zero science pool and no `tech_node_unlock` rows; the same world content created with `game_mode=science` reports only seeded-unlocked nodes available until science is spent. Switching mode is decided at world creation, not per-client.
- **Est:** M

### M5-T08 — Mode-aware client catalog gating in handshake and VAB read-model
- **Satisfies:** PROG-4, PROG-3
- **Depends on:** M5-T03, M5-T07
- **Intent:** Ensure clients present the parts catalog according to the authoritative mode + unlocked set — all parts in Sandbox, only unlocked parts in Science — so the shared progression is visibly consistent across all four players.
- **Touches:** Client `ProgramReadModel` (from M5-T03), client parts-catalog/VAB availability filter, wire `ProgramStateDto.gameMode` + unlocked set.
- **Steps:**
  1. In the client, resolve part availability from `ProgramReadModel`: Sandbox → all parts shown enabled; Science/Career → only parts whose owning node is in the unlocked set.
  2. Re-evaluate the catalog when a program-state update arrives (e.g. a node just got unlocked) so newly unlocked parts appear without reconnect.
  3. Keep the client purely presentational — availability is advisory UI; the server still authoritatively rejects launches/edits using locked parts (Art. 1).
- **Acceptance:** In a Science world, Client B's parts catalog shows a part as locked; after Client A spends science to unlock its node, Client B's catalog shows that part as available within one snapshot without reconnecting. In a Sandbox world all parts show available on join.
- **Est:** S
