# M1 — Physics bubble + prediction — code-ready tasks

> Expanded from roadmap.md slices 1.1–1.5. Each ticket cites the spec requirement(s) it satisfies. Terms in *italics* are in CONTEXT.md.

## Slice 1.1 — Bubble + floating origin (resolve P-4 first)

### M1-T01 — Define the bubble data model and registry
- **Satisfies:** PHYS-1
- **Depends on:** —
- **Intent:** Introduce the *physics bubble* as a first-class sim-core object so the server can hold several at once, each isolated in its own local coordinate space.
- **Touches:** `PhysicsBubble` (id, member vessel ids, `FloatingOrigin`, local-space rigidbody set), `BubbleRegistry` (id → bubble), sim-core scheduler hook.
- **Steps:**
  1. Define `PhysicsBubble` holding a stable `BubbleId`, the set of member *vessel* ids, a `FloatingOrigin` (world-space anchor in double precision), and a handle to its isolated physics scene.
  2. Create a `BubbleRegistry` owned by the sim core that creates/looks-up/destroys bubbles by id; assert ids are unique and stable for a bubble's lifetime.
  3. Allocate each bubble its own isolated Unity physics scene (`PhysicsScene` via `SceneManager.CreateScene` with `LocalPhysicsMode.Physics3D`) so bubbles never collide cross-cluster.
  4. Register every live bubble with the fixed-60 Hz scheduler (ADR-0006) so each is stepped independently per tick.
- **Acceptance:** Unit test creates 3 bubbles, registers 1 vessel in each, steps the scheduler 60 ticks; each bubble's physics scene reports exactly its own member colliders and zero cross-scene contacts.
- **Est:** M

### M1-T02 — Cluster detection: assign vessels to bubbles by proximity
- **Satisfies:** PHYS-1, PHYS-2
- **Depends on:** M1-T01
- **Intent:** Decide which vessels belong to the same bubble from world-space proximity, so co-located vessels share one simulation and far-apart ones run in parallel bubbles.
- **Touches:** `BubbleManager` (clustering pass), `BubbleRegistry`.
- **Steps:**
  1. Each tick, run a clustering pass over all active-physics candidate vessels using a fixed physics-range radius `R_phys` (config constant; default 2.5 km) in double-precision world coordinates.
  2. Group vessels whose pairwise world distance ≤ `R_phys` into the same cluster (union-find over the proximity graph).
  3. Map each cluster to a bubble: reuse the existing bubble if the cluster's members already share one; otherwise request creation via `BubbleRegistry`.
  4. Emit `BubbleFormed` / `BubbleJoined` events for downstream (promotion, replication) consumers.
- **Acceptance:** Test places two vessels 5 km apart → two bubbles; moves them to 2 km apart → clustering pass produces one bubble containing both within one tick of crossing `R_phys`.
- **Est:** M

### M1-T03 — Floating-origin rebasing within a bubble
- **Satisfies:** PHYS-1 (Art. 3)
- **Depends on:** M1-T01
- **Blocked by:** open item P-4 (floating-origin rebasing strategy)
- **Intent:** Keep every bubble's local coordinates near zero so 32-bit float physics stays precise at interplanetary world distances (prevent the "kraken", ADR-0003).
- **Steps:**
  1. Maintain each bubble's `FloatingOrigin` as a double-precision world anchor; local transforms are `world − origin`.
  2. Each tick compute the bubble centroid in local space; when its magnitude exceeds a rebase threshold (default 5 km), shift `FloatingOrigin` to the current centroid and subtract the delta from every member rigidbody's local position.
  3. Preserve velocities and angular state unchanged across the rebase (translation-only shift; no rotation of the frame).
  4. Per P-4: document and implement the chosen rule for which anchor a *merged* bubble adopts and how a *split* bubble re-anchors each fragment (record decision back into plan.md P-4).
- **Touches:** `FloatingOrigin`, `BubbleManager` rebase pass.
- **Acceptance:** Test places a vessel at 1×10^9 m world distance inside a bubble, runs 600 ticks of free coast; local-space position stays < 5 km from origin and integrated trajectory matches the double-precision reference within 1 cm/s velocity error (no precision artifacts / jitter).
- **Est:** L

### M1-T04 — Promotion: on-rails → active-physics on player approach/load
- **Satisfies:** PHYS-2
- **Depends on:** M1-T02, M1-T03
- **Intent:** Switch a vessel from cheap analytic propagation into a rigid-body bubble exactly when a player loads it or it closes within physics range of another active vessel.
- **Touches:** `PromotionController`, `BubbleManager`, `OnRailsPropagator` (read handoff), `RigidVesselBody`.
- **Steps:**
  1. Define promotion triggers (PHYS-2): a player loads/approaches the vessel within `R_phys`, OR two vessels close within `R_phys` (reuse M1-T02 clustering).
  2. On trigger, sample the *on-rails vessel*'s closed-form `position(game-time)` and orbital velocity at the current *master clock* tick (ORBIT-2 propagator).
  3. Instantiate a `RigidVesselBody` in the target bubble's physics scene, seeding its local transform (world − `FloatingOrigin`) and linear/angular velocity from the sampled state.
  4. Mark the vessel `ActivePhysics` and remove it from the on-rails propagation set so it is no longer analytically advanced.
- **Acceptance:** Test starts a vessel on-rails, moves a player within `R_phys`; within one tick the vessel is `ActivePhysics`, lives in a bubble, and its first-tick world position/velocity equal the analytic sample within 1 mm / 1 mm/s.
- **Est:** M

### M1-T05 — Demotion: active-physics → on-rails when warp-safe and unattended
- **Satisfies:** PHYS-3
- **Depends on:** M1-T04
- **Intent:** Return an unattended, *warp-safe* vessel to analytic propagation so the server stays cheap and the vessel rejoins the living universe on rails (ADR-0002/0003).
- **Touches:** `PromotionController` (demotion path), `WarpSafeEvaluator`, `OnRailsPropagator`, `OrbitFitter`.
- **Steps:**
  1. Each tick evaluate `WarpSafeEvaluator` for active vessels with no present player: warp-safe = stable orbit or landed, no active thrust, not under atmospheric forces (CONTEXT: warp-safe state).
  2. When warp-safe AND unattended, fit a closed-form *orbit* from the vessel's current rigid-body position+velocity around its current *SOI* body (`OrbitFitter` → orbital elements).
  3. Remove the `RigidVesselBody` from the bubble, mark the vessel `OnRails`, and add it back to the on-rails propagation set synced to the *master clock*.
  4. Destroy the bubble via `BubbleRegistry` if it now has zero members.
- **Acceptance:** Test promotes a coasting orbital vessel, removes the player; within one tick it demotes, and its analytic `position(game-time)` is continuous with the last rigid-body position/velocity within 1 cm / 1 cm/s (no state jump across the switch).
- **Est:** M

## Slice 1.2 — Active rigid-body flight (resolve P-1 first)

### M1-T06 — Rigid-vessel integrator: gravity in the bubble
- **Satisfies:** PHYS-4
- **Depends on:** M1-T01, M1-T03
- **Intent:** Step each active *rigid vessel* under its single SOI body's gravity at the fixed 60 Hz tick, as one rigid body with no inter-part flex (ADR-0005/0006).
- **Touches:** `RigidVesselBody`, `BubbleIntegrator`, patched-conics gravity source.
- **Steps:**
  1. Represent each active vessel as one rigid body (mass, inertia tensor from the *part tree*; split into separate bodies only at decouplers/stages per ADR-0005).
  2. Each fixed tick, apply gravitational acceleration from the vessel's current SOI body (`g = -μ r̂ / |r|²`, single-body per ORBIT-1) at the bubble's local frame.
  3. Integrate with a fixed-`dt` symplectic/semi-implicit step (no variable `dt`, no frame coupling, ADR-0006).
  4. Forbid inter-part soft joints; only *articulation points* may move (covered by docking/robotics tickets).
- **Acceptance:** Test drops a vessel from a circular orbit's apoapsis with gravity only; after 600 ticks its bubble trajectory matches the closed-form Kepler reference within 0.1% position error.
- **Est:** M

### M1-T07 — Thrust + mass-flow producing measured delta-v
- **Satisfies:** PHYS-4
- **Depends on:** M1-T06
- **Intent:** Apply engine thrust and propellant mass depletion to the rigid vessel so a burn yields the expected delta-v.
- **Touches:** `RigidVesselBody`, `PropulsionSystem`, `BubbleIntegrator`.
- **Steps:**
  1. Model each active engine's thrust vector in vessel-local space and apply it as a force at the engine's mount point each tick.
  2. Deplete propellant by mass flow `ṁ = F / (Isp · g₀)` and update vessel mass + inertia tensor each tick.
  3. Integrate thrust together with gravity in the same fixed-`dt` step (M1-T06).
- **Acceptance:** Test burns a known stage to depletion; measured delta-v equals the Tsiolkovsky prediction `Δv = Isp·g₀·ln(m0/m1)` within 1%.
- **Est:** M

### M1-T08 — Pilot input channel: throttle + attitude routed to server, applied authoritatively
- **Satisfies:** NET-1, CREW-1 (partial)
- **Depends on:** M1-T07
- **Blocked by:** open item P-1 (snapshot wire format) — reuse its agreed channel/codec for inputs
- **Intent:** Carry the *Pilot* station's throttle and attitude commands to the authoritative server, which applies them to the rigid vessel (server is sole authority, NET-1).
- **Touches:** `InputChannel`, `StationInputRouter`, `PilotStation`, `RigidVesselBody`.
- **Steps:**
  1. Define a per-station, per-tick input message: throttle [0..1] and attitude command (pitch/yaw/roll), tagged with the client tick number for later reconciliation.
  2. Route inputs through `StationInputRouter` to the Pilot station's disjoint system set (attitude + throttle only — CREW-1), rejecting fields not owned by Pilot.
  3. On the server, buffer inputs per source tick and apply the authoritative input for the current sim tick to the `RigidVesselBody` (throttle → engines, attitude → reaction control / SAS torque).
  4. Use the P-1 wire format for the input channel codec; flag if format is unresolved.
- **Acceptance:** Test feeds a scripted throttle/attitude sequence over the channel; server-applied per-tick inputs match the sent sequence exactly, and a staging/engineer-owned field sent on the Pilot channel is rejected (logged).
- **Est:** M

### M1-T09 — Discrete structural failure at a load threshold
- **Satisfies:** PHYS-6
- **Depends on:** M1-T06
- **Intent:** Break a joint/decoupler as a discrete event when load exceeds its threshold — no continuous soft-body flex (ADR-0005).
- **Touches:** `StructuralFailureSystem`, `ArticulationPoint`, `RigidVesselBody`, `BubbleManager` (re-split).
- **Steps:**
  1. Each tick compute the load (force/torque) carried across each decoupler and articulation joint of the rigid vessel.
  2. When load exceeds the joint's configured threshold, emit a discrete `StructuralFailure` event (no gradual flex, no soft spring).
  3. Split the affected subtree into a new rigid body within the same bubble, inheriting the parent's velocity/angular state at the break point.
  4. Re-register the resulting bodies with `BubbleManager` (the bubble may now contain two vessels).
- **Acceptance:** Test applies a ramping load across a joint; below threshold nothing happens, at threshold exactly one `StructuralFailure` event fires and the vessel becomes two rigid bodies in one tick — no intermediate flex frames observed.
- **Est:** M

## Slice 1.3 — Prediction + reconciliation

### M1-T10 — Client-side prediction of the controlled vessel
- **Satisfies:** NET-2, NET-6
- **Depends on:** M1-T08
- **Intent:** Let the *Pilot*'s client simulate its own controlled vessel locally from inputs so the stick has zero perceived input lag (applied only to the locally-controlled vessel).
- **Touches:** `ClientPredictor`, `InputBuffer`, shared `BubbleIntegrator` (run client-side), `RigidVesselBody`.
- **Steps:**
  1. Run the same fixed-step `BubbleIntegrator` on the client for the controlled vessel only (identical sim core, ADR-0006), so prediction and server use one code path.
  2. On each input, immediately apply it locally and advance the predicted state; render from the predicted state with no wait for server ack.
  3. Store every input in an `InputBuffer` keyed by client tick, retaining all inputs not yet acked by the server.
  4. Apply prediction strictly to the locally-controlled vessel; everything else stays on interpolation (NET-4, M1-T13 boundary).
- **Acceptance:** Test with 80 ms simulated RTT injects a control input; the rendered controlled-vessel state reflects the input on the same client frame (perceived input lag ≈ 0), and only the controlled vessel is predicted.
- **Est:** M

### M1-T11 — Server reconciliation: smoothed correction / hard snap on large desync
- **Satisfies:** NET-3
- **Depends on:** M1-T10
- **Intent:** Correct the client's predicted state toward *authoritative state* when they diverge — smoothing small corrections, hard-snapping only on large desync (ADR-0006 reconciliation budget).
- **Touches:** `ClientReconciler`, `ClientPredictor`, `InputBuffer`, snapshot consumer.
- **Steps:**
  1. On each authoritative snapshot, read the last-acked server tick and the server's state at that tick for the controlled vessel.
  2. Reset the client's controlled vessel to the acked authoritative state, then replay all buffered inputs from the last-acked tick forward to the present client tick (`InputBuffer`).
  3. Compute divergence between pre-replay predicted state and the re-derived state; if below a position/velocity threshold, smooth the visual correction over a few frames; if above, hard-snap immediately.
  4. Discard inputs at or before the acked tick from the buffer.
- **Acceptance:** Test injects a sub-threshold server divergence → correction is applied with no single-frame visible pop (interpolated over N frames); injects an above-threshold divergence → state hard-snaps within one frame. Both assert final state converges to authoritative within tolerance.
- **Est:** L

### M1-T12 — Graceful degradation beyond 150 ms RTT
- **Satisfies:** NET-6
- **Depends on:** M1-T11
- **Intent:** Keep piloting smooth at ≤150 ms RTT and degrade gracefully (not catastrophically) beyond it.
- **Touches:** `ClientReconciler`, `InputBuffer`, `LatencyMonitor`.
- **Steps:**
  1. Track measured RTT in a `LatencyMonitor`.
  2. Size the input/reconciliation buffer from measured RTT so replay covers the in-flight window without overflow.
  3. Beyond 150 ms, widen reconciliation smoothing windows and cap correction magnitude per frame so corrections stay visually continuous rather than snapping repeatedly.
- **Acceptance:** Test at 150 ms RTT asserts piloting smoothness metric (no hard snaps under nominal divergence); at 300 ms RTT asserts the client still converges to authoritative state with corrections bounded per frame (no oscillation, no runaway).
- **Est:** M

### M1-T13 — Interpolate all non-controlled vessels (prediction boundary)
- **Satisfies:** NET-4, NET-2
- **Depends on:** M1-T10
- **Intent:** Display every vessel the client does not control via *interpolation* of authoritative snapshots with a small delay — never predicted — enforcing the prediction boundary.
- **Touches:** `SnapshotInterpolator`, `ClientPredictor` (exclusion), replication consumer.
- **Steps:**
  1. Buffer incoming authoritative snapshots per non-controlled vessel with timestamps.
  2. Render each non-controlled vessel at `now − interpolation_delay`, interpolating transform + velocity between the two bracketing snapshots.
  3. Assert the predictor never touches non-controlled vessels (prediction applied only to the locally-controlled vessel, NET-2).
- **Acceptance:** Test renders one controlled (predicted) and one non-controlled (interpolated) vessel; the controlled responds to local input same-frame, the non-controlled trails by exactly the interpolation delay and is never advanced by the predictor.
- **Est:** S

## Slice 1.4 — Suspension lifecycle

### M1-T14 — Suspend a non-warp-safe vessel on last-leave
- **Satisfies:** SUSP-3
- **Depends on:** M1-T05
- **Intent:** When the last present player leaves a vessel that is not *warp-safe*, snapshot it and pause its *vessel clock* rather than fake an orbit or burn CPU (ADR-0002).
- **Touches:** `SuspensionController`, `WarpSafeEvaluator`, `VesselClock`, `VesselSnapshot`, `BubbleManager`.
- **Steps:**
  1. On last-player-leave, evaluate `WarpSafeEvaluator`; if NOT warp-safe (in atmosphere, under thrust, mid-docking), take the suspend path.
  2. Capture a `VesselSnapshot`: full rigid-body state (local transform, linear/angular velocity), resources, staging/articulation state, and the current *vessel clock* value.
  3. Mark the vessel `Suspended`, pause its `VesselClock`, and exclude it from *game-time* advance (no retro-sim, no rail-snap — Art. 4).
  4. Remove its `RigidVesselBody` from the bubble; destroy the bubble if empty.
- **Acceptance:** Test puts a vessel under thrust (not warp-safe), removes the last player; vessel becomes `Suspended`, a snapshot exists, its vessel clock value is frozen, and advancing the master clock 60 s leaves the vessel clock unchanged.
- **Est:** M

### M1-T15 — Resume a suspended vessel from snapshot on reload (no retro-sim)
- **Satisfies:** SUSP-4
- **Depends on:** M1-T14
- **Intent:** When a player next loads a *suspended vessel*, resume it from its snapshot exactly — no retro-simulation, no rail-snap (ADR-0002, Art. 4).
- **Touches:** `SuspensionController` (resume path), `VesselSnapshot`, `VesselClock`, `PromotionController`, `BubbleManager`.
- **Steps:**
  1. On player load of a `Suspended` vessel, create/join a bubble and instantiate a `RigidVesselBody` seeded byte-for-byte from the `VesselSnapshot` (transform, velocities, resources, staging).
  2. Resume the `VesselClock` from its frozen value (the vessel clock may lag the master clock — CONTEXT: vessel clock).
  3. Mark the vessel `ActivePhysics`; do NOT advance it to cover the elapsed master-clock interval (no retro-sim).
- **Acceptance:** Test suspends a vessel, advances master clock 1 h, reloads it; the first active tick's state equals the snapshot within 1 mm / 1 mm/s, and the vessel clock resumes from the frozen value (lagging the master clock by ~1 h).
- **Est:** M

### M1-T16 — Demote (not suspend) when warp-safe on last-leave
- **Satisfies:** SUSP-2
- **Depends on:** M1-T05
- **Intent:** When the last present player leaves a *warp-safe* vessel, route it to demotion (on-rails), not suspension — the common case for a coasting orbiter.
- **Touches:** `SuspensionController` (branch), `WarpSafeEvaluator`, `PromotionController` (demotion path from M1-T05).
- **Steps:**
  1. On last-player-leave, branch on `WarpSafeEvaluator`: warp-safe → demotion path (M1-T05); not warp-safe → suspend path (M1-T14).
  2. For the warp-safe branch, invoke demotion: fit orbital elements, mark `OnRails`, sync to the master clock.
  3. Assert no `VesselSnapshot` is taken and the vessel clock stays synced to the master clock (it did not pause).
- **Acceptance:** Test puts a vessel in a stable orbit (warp-safe), removes the last player; vessel becomes `OnRails` (not `Suspended`), no snapshot is taken, and its analytic `position(game-time)` continues synced to the master clock.
- **Est:** S

## Slice 1.5 — Docking

### M1-T17 — Docking-port proximity detection and latch within tolerance
- **Satisfies:** PHYS-5
- **Depends on:** M1-T02, M1-T06
- **Intent:** Detect when two docking ports of two vessels in the *same physics bubble* are within alignment tolerance and latch them.
- **Touches:** `DockingPort` (*articulation point*), `DockingSystem`, `BubbleManager`.
- **Steps:**
  1. Model each docking port as an *articulation point* with a capture frame (position + axis) on its vessel.
  2. Each tick, for ports of distinct vessels in the same bubble, test capture tolerance: relative distance ≤ `d_cap`, axis misalignment ≤ `θ_cap`, closing speed ≤ `v_cap` (config constants).
  3. When all tolerances are met, fire a `DockLatched` event for the port pair.
- **Acceptance:** Test brings two ports together within `d_cap`/`θ_cap`/`v_cap`; exactly one `DockLatched` event fires the tick tolerances are met; a near-miss outside tolerance fires none.
- **Est:** M

### M1-T18 — Join two latched vessels into one rigid vessel
- **Satisfies:** PHYS-5
- **Depends on:** M1-T17
- **Intent:** On latch, merge the two vessels into a single *rigid vessel* (joined at the docking-port articulation point), recomputing combined mass and inertia.
- **Touches:** `DockingSystem`, `RigidVesselBody`, part-tree merge, `BubbleManager`.
- **Steps:**
  1. On `DockLatched`, merge the two *part trees* at the port pair, making one vessel's tree a child subtree of the other under the docking articulation.
  2. Combine into a single `RigidVesselBody`: summed mass, recomputed center of mass and inertia tensor, conserving total linear and angular momentum at the join.
  3. Deregister the absorbed vessel id and keep the merged vessel as one bubble member.
- **Acceptance:** Test docks two equal-mass vessels with known velocities; the merged vessel's mass, COM, and post-join momentum equal the conservation-of-momentum prediction within 1%, and it is a single rigid body (one transform replicated).
- **Est:** M

### M1-T19 — No authority handoff at contact (shared bubble before latch)
- **Satisfies:** PHYS-5 (Art. 1)
- **Depends on:** M1-T02, M1-T17
- **Intent:** Guarantee both vessels already share one bubble (one machine, the server) before contact, so docking needs no cross-machine authority handoff and shows no state jump (ADR-0001/0003).
- **Touches:** `BubbleManager`, `DockingSystem`, replication snapshot consumer.
- **Steps:**
  1. Assert that, by clustering (M1-T02), two converging vessels join the same bubble at `R_phys` — well before the much smaller capture distance `d_cap`.
  2. Verify the latch (M1-T18) operates entirely within that single server-authoritative bubble; no authority transfer message is sent or required.
  3. Add a regression check that no discontinuity is injected into either vessel's authoritative state at the moment of latch.
- **Acceptance:** Test logs each vessel's bubble id from approach through latch: both share one bubble id from the moment they cross `R_phys` onward; at latch, neither vessel's authoritative position/velocity jumps (delta ≤ integrator noise), and zero authority-handoff messages are emitted.
- **Est:** S
