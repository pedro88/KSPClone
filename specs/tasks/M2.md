## Slice 2.1 â€” Station occupancy

### M2-T01 â€” Define the Station model and per-vessel station set
- **Satisfies:** CREW-1
- **Depends on:** â€”
- **Intent:** Give every *vessel* a fixed, server-owned set of *stations* (Pilot, Engineer, Navigator) so occupancy and input routing have a stable target. Authoritative state only â€” no client ownership (Art. 1).
- **Touches:** `Station` (struct/enum `StationKind` + occupancy field), `VesselStations` (per-vessel container), `CrewState` on the vessel's authoritative record.
- **Steps:**
  1. Add `StationKind { Pilot, Engineer, Navigator }` and a `Station` record holding `StationKind kind` and `PlayerId? occupant` (null = unoccupied).
  2. Attach a `VesselStations` collection to each authoritative vessel record, initialized with exactly one Station per `StationKind`, all unoccupied at spawn.
  3. Ensure `VesselStations` lives in *authoritative state* on the server and is included in the snapshot the client already receives (M0 handshake / NET path).
- **Acceptance:** Unit test: a freshly spawned vessel exposes exactly 3 stations (one per `StationKind`), all with `occupant == null`. Server log on spawn lists the three stations.
- **Est:** S

### M2-T02 â€” Occupy / vacate / hot-swap a station (server command + result)
- **Satisfies:** CREW-2
- **Depends on:** M2-T01
- **Intent:** Let a player *occupy* at most one *station* at a time on a vessel and hot-swap to another free station without an intermediate "stand-up" step. All transitions are authoritative.
- **Touches:** `StationOccupancyService` (server), `OccupyStationCommand { VesselId, StationKind }`, `VacateStationCommand`, `StationOccupancyChanged` event.
- **Steps:**
  1. Implement `Occupy(playerId, vesselId, kind)`: if the player already occupies a station on that vessel, atomically vacate the old and seat the new (hot-swap); otherwise seat into the requested free station.
  2. Enforce the at-most-one-station-per-player-per-vessel invariant inside the service (a player cannot hold two stations on the same vessel).
  3. Implement `Vacate(playerId, vesselId)` clearing that player's `occupant` and emitting `StationOccupancyChanged`.
  4. Apply commands on the fixed-step server tick so ordering is deterministic (Art. 2); broadcast the resulting occupancy in the snapshot.
- **Acceptance:** Test sequence: player occupies Pilot (success); same player occupies Navigator on same vessel â†’ Pilot becomes unoccupied and Navigator occupant == player (hot-swap in one step). After Vacate, all that player's stations are unoccupied.
- **Est:** M

### M2-T03 â€” Single-occupant enforcement under concurrent claims
- **Satisfies:** CREW-3
- **Depends on:** M2-T02
- **Intent:** When two players claim the same *station*, grant it to exactly one and refuse the other deterministically (no shared seat, Art. 6).
- **Touches:** `StationOccupancyService`, `OccupyResult { Granted | Refused(reason) }`.
- **Steps:**
  1. Resolve competing `OccupyStationCommand`s for the same station in server-tick arrival order; first to apply wins, later ones see a non-null occupant and return `Refused(StationTaken)`.
  2. Return an explicit `OccupyResult` to each requesting client (grant vs refusal), not silent failure.
  3. Log refusals with vessel, station, winner playerId, loser playerId.
- **Acceptance:** Test: two `Occupy(Pilot)` commands for the same vessel applied in the same tick batch â†’ exactly one `Granted`, one `Refused(StationTaken)`; station occupant equals the winner. Assertion holds regardless of submission order.
- **Est:** S

## Slice 2.2 â€” Disjoint input routing

### M2-T04 â€” Declare the disjoint system set owned by each station
- **Satisfies:** CREW-1
- **Depends on:** M2-T01
- **Intent:** Pin down, in one place, which controllable systems each *station* owns so the partition is by construction and provably disjoint (Art. 6): Pilot = attitude + throttle; Engineer = staging + resources + power + abort; Navigator = maneuver nodes + map/transfer planning.
- **Touches:** `ControllableSystem` (enum), `StationSystemMap` (static map `StationKind â†’ set<ControllableSystem>`), a startup self-check.
- **Steps:**
  1. Define `ControllableSystem { Attitude, Throttle, Staging, Resources, Power, Abort, ManeuverNode, MapPlanning }`.
  2. Define `StationSystemMap` assigning each system to exactly one `StationKind` per the partition above.
  3. Add a startup assertion that the union of all stations' system sets covers every `ControllableSystem` and that the sets are pairwise disjoint (no system owned by two stations).
- **Acceptance:** Unit test: for every pair of stations, intersection of their system sets is empty; union equals the full `ControllableSystem` set. Startup self-check passes; deliberately duplicating a system across two stations fails the check.
- **Est:** S

### M2-T05 â€” Route per-station input on the authoritative server, rejecting cross-station inputs
- **Satisfies:** CREW-1
- **Depends on:** M2-T02, M2-T04
- **Intent:** Extend the M1 input channel (T20) so each input names its target *station*; the server applies it only if the sender occupies that station and the input targets a system that station owns. Pilot inputs can never touch staging and vice-versa.
- **Touches:** `InputRouter` (server), `StationInput { PlayerId, VesselId, StationKind, ControllableSystem, payload }`, extends the M1 input channel.
- **Steps:**
  1. Tag every input message with its `StationKind` and `ControllableSystem`.
  2. In `InputRouter`, on each input: verify `occupant(vessel, stationKind) == sender` and `StationSystemMap[stationKind].Contains(system)`; otherwise drop and log a rejection.
  3. Dispatch accepted inputs to the owning subsystem handler (attitude/throttle for Pilot, etc.) on the fixed-step tick.
- **Acceptance:** Test: a player seated at Pilot sends a Staging input â†’ rejected and logged (Pilot does not own Staging); the same player's Throttle input â†’ applied. A non-occupant's Attitude input for the vessel â†’ rejected.
- **Est:** M

### M2-T06 â€” Concurrent multi-station inputs applied conflict-free in one tick
- **Satisfies:** CREW-1, CREW-3 *(Art. 6)*
- **Depends on:** M2-T05
- **Intent:** Prove three occupied *stations* on one vessel can act in the same fixed-step tick with no contention, because their target systems are disjoint by construction.
- **Touches:** `InputRouter`, fixed-step tick integration (Art. 2), `VesselControlState`.
- **Steps:**
  1. Within a single tick, accept Pilot (attitude/throttle), Engineer (staging), and Navigator (node) inputs for the same vessel and apply each to its owning subsystem.
  2. Confirm no input handler writes a field owned by another station (rely on the disjoint map; add a write-guard or assertion in `VesselControlState` if feasible).
- **Acceptance:** Integration test: with Pilot/Engineer/Navigator all occupied, submit one input from each in the same tick; after the tick, attitude/throttle reflect Pilot input, stage index reflects Engineer input, and the maneuver node reflects Navigator input â€” all three present, none overwriting another.
- **Est:** M

## Slice 2.3 â€” Automation fallback

### M2-T07 â€” Automation fallback framework driving unoccupied stations
- **Satisfies:** CREW-4
- **Depends on:** M2-T02, M2-T05
- **Intent:** While a *station* is unoccupied, the server runs that station's *automation fallback* as the input source for its owned systems, so a short-crewed or solo vessel stays flyable but degraded (always dumber than a human).
- **Touches:** `AutomationFallback` (per-`StationKind` strategy interface), `StationDriver` (selects human input vs automation each tick), `InputRouter`.
- **Steps:**
  1. Define `IStationAutomation.Tick(vesselState) -> StationInput?` with a no-op default per station.
  2. In `StationDriver`, each tick: if `occupant == null`, source that station's systems from its `IStationAutomation`; if occupied, source from human input only (automation suppressed).
  3. Route automation outputs through the same `InputRouter` system-ownership checks as human input (no automation may touch another station's systems).
- **Acceptance:** Test: vacate Pilot â†’ `StationDriver` reports Pilot's systems sourced from automation; occupy Pilot â†’ source switches to human input, automation suppressed within the same tick boundary.
- **Est:** M

### M2-T08 â€” Pilot automation: SAS attitude hold
- **Satisfies:** CREW-4
- **Depends on:** M2-T07
- **Intent:** An empty Pilot *station* holds the vessel's current attitude (SAS-style) so an uncrewed Pilot does not tumble the vessel â€” dumber than a human (hold only, no maneuvers).
- **Touches:** `PilotAutomation : IStationAutomation`, attitude subsystem.
- **Steps:**
  1. On first tick unoccupied, capture the vessel's current orientation as the hold target.
  2. Each tick, emit attitude inputs that null out angular rate toward the captured target (proportional/damped); emit zero throttle change.
  3. Reset the hold target when the station becomes occupied then vacated again.
- **Acceptance:** Test: spin up a vessel's angular velocity, vacate Pilot â†’ within N ticks angular rate decays below threshold and orientation stabilizes near the captured target; throttle is untouched (Engineer's/Pilot-owned throttle stays at last value, no maneuver issued).
- **Est:** M

### M2-T09 â€” Engineer automation: auto-stage on flameout
- **Satisfies:** CREW-4
- **Depends on:** M2-T07
- **Intent:** An empty Engineer *station* advances staging when the active stage flames out (no thrust / fuel exhausted), so a solo pilot is not stranded on a dead stage â€” dumber than a human (flameout-triggered only).
- **Touches:** `EngineerAutomation : IStationAutomation`, staging subsystem, flameout detection.
- **Steps:**
  1. Each tick, detect flameout of the active stage (all engines in stage producing zero thrust while a next stage exists).
  2. On flameout, emit one Staging input to advance to the next stage; debounce so a single flameout fires exactly one stage event.
  3. Do nothing for resources/power/abort (those Engineer systems have no fallback in this slice â€” leave at last state).
- **Acceptance:** Test: with Engineer unoccupied, force the active stage to flameout â†’ exactly one stage-advance event fires and stage index increments by one; no further staging while the new stage burns. With Engineer occupied, automation does not auto-stage.
- **Est:** M

## Slice 2.4 â€” Disconnect handling

### M2-T10 â€” Disconnect vacates all of a player's stations to automation, no reserved seat
- **Satisfies:** CREW-5
- **Depends on:** M2-T02, M2-T07
- **Intent:** When an occupying player disconnects, the server vacates every *station* they held (across any vessel), each falling back to *automation*, and reserves no seat (Art. 1; local-crew authority handed back to fallback, Art. 5).
- **Touches:** `StationOccupancyService`, disconnect/session hook, `StationDriver`, `AutomationFallback`.
- **Steps:**
  1. Hook the server's player-disconnect event to enumerate all stations whose `occupant == disconnectedPlayer` and vacate each (clear occupant, emit `StationOccupancyChanged`).
  2. Confirm `StationDriver` immediately sources each vacated station from its automation on the next tick (reuse M2-T07).
  3. Store no per-player seat reservation anywhere; vacated stations are plain free stations.
- **Acceptance:** Test: player occupies Pilot, then disconnects â†’ Pilot `occupant == null` within one tick, Pilot systems sourced from automation, and no reservation record exists for that player.
- **Est:** S

### M2-T11 â€” Reconnect can re-occupy any free station (no priority for prior occupant)
- **Satisfies:** CREW-5, CREW-2
- **Depends on:** M2-T10, M2-T03
- **Intent:** A reconnecting player may *occupy* any currently free *station* with no precedence from their pre-disconnect seat; if another player took it meanwhile, the standard single-occupant refusal applies.
- **Touches:** `StationOccupancyService`, reconnect/session path, `OccupyResult`.
- **Steps:**
  1. On reconnect, grant the player no automatic re-seating; they must issue a fresh `OccupyStationCommand`.
  2. Route that command through the normal occupy path (M2-T02) and single-occupant enforcement (M2-T03) â€” the previously-held station gets no special priority.
- **Acceptance:** Test: player holds Pilot, disconnects (Pilot frees), a second player takes Pilot; first player reconnects and requests Pilot â†’ `Refused(StationTaken)`; requests Navigator (free) â†’ `Granted`. A reconnect into a still-free prior station succeeds only via an explicit command, with no reservation shortcut.
- **Est:** S
